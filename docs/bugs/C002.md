## Delta Analysis

### Here-String Sources `[shellcheck-quirk]`

**Bucket:** ShellCheck-only  
**Impact:** 2 locations / 2 scripts

ShellCheck reports `SC1090` for `source /dev/stdin <<< ...`, but the source target itself is the fixed path `/dev/stdin`. Shuck stays silent here because the file path is not runtime-built, so this looks like a ShellCheck false positive rather than a shuck bug.

**Samples:**
- `tteck__Proxmox__install__cronicle-install.sh:28` - `source /dev/stdin <<< "$FUNCTIONS_FILE_PATH"`
- `tteck__Proxmox__install__nginxproxymanager-install.sh:58` - `source /dev/stdin <<<"$FUNCTIONS_FILE_PATH"`

### Project-Closure Source Paths `[shuck-correct]`

**Bucket:** Shuck-only  
**Impact:** 354 locations / 251 scripts

Most of the sample is shuck warning on source paths that are composed at runtime from project-local variables, helper directories, or repo conventions. ShellCheck stays quiet in these cases because it can follow more of the source graph, because a directive suppresses the warning, or because the script lives in a project-closure pattern ShellCheck does not model the same way.

This is the dominant divergence for C002. It reads more like an intentional policy difference than a narrow implementation bug.

**Samples:**
- `rvm__rvm__scripts__pkg:9-11` - `source "$rvm_scripts_path/base"` and related helper loads
- `HariSekhon__DevOps-Bash-tools__bin__run.sh:20-32` - helper loads through `"$srcdir/..."`
- `Bash-it__bash-it__completion__available__travis.completion.bash:12` - project-local helper sourcing
- `termux__termux-packages__scripts__run-docker.sh:215` - source path built from script-local state
