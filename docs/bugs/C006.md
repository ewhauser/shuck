## Oracle probe findings

### Any in-file name introduction suppresses `SC2154` `[probed]`

The generic binding suite produced one simple rule:

- `25/26` probe scripts were quiet.
- Only the pure no-binding case still emitted `SC2154`.

Quiet cases included all of the representative binding kinds Shuck currently collapses:

- plain assignment
- assignment in another function
- assignment in a dead branch
- `local`, `declare`, `typeset`, `readonly`, `export`, `declare -g`
- name-only declarations and initialized declarations
- nameref
- `${name:=...}`
- `+=`
- array assignment
- arithmetic assignment
- loop variable
- `read`, `mapfile`, `printf -v`, `getopts`

This strongly supports keeping the current broad "any in-file defining binding suppresses C006"
policy.

### ShellCheck reports the first occurrence of a name, not every repeat `[probed]`

The repetition probes were decisive:

- same-line repeats: one report
- later-line repeats: one report
- repeats in another function: one report
- two different names: one report per distinct name

A minimal `rvm`-shaped probe behaved the same way: the first `rvm_src_path` and first
`rvm_ruby_string` use were reported, later repeats were quiet.

This explains a large share of the current `rvm` tail. Many corpus entries currently counted as
`Shuck-only` are later repeats of a name that ShellCheck already reported earlier in the same
file.

### Source following is narrower than our previous working model `[probed]`

The source-following matrix split into two groups:

- Static relative and repo-relative-like literal paths can be followed, but only when ShellCheck
  runs from the target file's directory and the helper path is literal-followable.
- Under the large-corpus harness conditions, ShellCheck does not follow those helpers because it
  runs from the repo root and the relative path no longer resolves.

The following shapes stayed unresolved in every probe variant:

- one-variable static tail: `source "$rvm_path/scripts/rvm"`
- defaulted parameter path: `source "${rvm_scripts_path:-$rvm_path/scripts}/hook"`
- dynamic tail: `source "$rvm_scripts_path/functions/manage/${kind}"`

So the remaining `rvm_scripts_path`-style reports are not evidence of a hidden project database.
They are a mix of unresolved dynamic source paths and repeated later occurrences.

### Variable names themselves are not special `[probed]`

A bare minimal function using only the names below still triggered `SC2154` for all of them:

- `rvm_src_path`
- `rvm_ruby_string`
- `rvm_path`
- `pkgver`
- `pkgname`
- `wrksrc`
- `workdir`
- `formula`

That rules out the earlier working theory that ShellCheck might have a large built-in table of
`rvm` or packaging globals.

### Some current tails are construct-specific, not ambient-global behavior `[probed]`

Three important non-global buckets emerged:

- Array subscript index names:
  - ShellCheck stayed quiet on `__array_start` in `${args[$__array_start]}` / `args[$__array_start]=...`.
  - ShellCheck also stayed quiet on `$target` inside `${tools[$target]}` / `${!tools[$target]}`.
  - It may still warn on the base array name itself.
- Literal generated shell text:
  - `romkatv__powerlevel10k__gitstatus__build` is quiet under direct ShellCheck runs.
  - The relevant names (`formula`, `workdir`, `cflags`) live inside a quoted `command cat <<\END`
    template body.
- Escaped literal shell text:
  - `termux__termux-packages__packages__docbook-xsl__build.sh` is quiet under direct ShellCheck.
  - The reported `x` / `ver` sites are literal `\$x` / `\$ver` inside a generated maintainer script.

These are all Shuck-side false positives, but they are not ambient-contract problems.

### `void-packages` is mixed, not one reusable ambient bucket `[probed]`

Representative `void-packages` files fell into different classes:

- `common/environment/setup/install.sh` is quiet under direct ShellCheck, and the relevant sites
  carry explicit `SC2154` disables in the file itself.
- `common/hooks/do-fetch/00-distfiles.sh` still emits direct `SC2154` on first occurrences such as
  `pkgver`.
- `common/hooks/do-pkg/00-gen-pkg.sh` still emits direct `SC2154` on several framework names and
  unresolved helper reads.

So the remaining `void-packages` tail is not a single "add ambient globals" task. It is a mix of
directive handling, first-occurrence parity, and some real framework/file-entry questions.

That means the current raw `rvm` `Shuck-only` bucket is mostly:

- later repeats of the same name after ShellCheck's first report
- source-path occurrences that ShellCheck cannot follow under corpus conditions
- array-subscript/index cases such as `__array_start`
- some plain location/occurrence selection differences within the same file

It is not strong evidence that we need a repo-specific `rvm` global-provider table.

### `powerlevel10k` `[shuck-fix]`

This remains a clean inert-template problem.

- The current Shuck reports come from literal generated shell text in the quoted build template.

### `termux` `[shuck-fix]`

The representative `termux` tail split cleanly:

Again, not an ambient-contract problem.

### `void-packages` `[mixed]`

This family now needs a narrower audit:

Do not treat the whole family as one ambient-contract task.

## Concrete plan

1. Add first-occurrence parity for `C006`.
   - Report only the first reportable unresolved use of a given name per file, matching the
     repeated-use behavior shown by the probe suite.
   - This is the highest-leverage generic reducer for the current `rvm` and `void-packages` tail.

2. Stop reporting unresolved names that appear only in array-subscript index / indirect-subscript
   positions.
   - Target cases like `__array_start` and `$target`.
   - Treat this as `SC2154` parity work, not an ambient-contract task.

3. Finish inert generated-shell and escaped-dollar handling.
   - Cover quoted heredoc template bodies such as `command cat <<\END`.
   - Cover literal maintainer-script output containing escaped shell placeholders like `\$x`.

4. Re-audit the remaining source-path cases after steps 1-3.
   - Do not add `rvm`-specific ambient contracts yet.
   - The probe pass only justifies future source-following work for literal helper paths, and even
     that behavior depends on cwd and `-x`, which the large-corpus harness does not use.

5. Keep framework work narrow and evidence-driven.
   - Re-check `void-packages` helpers individually with direct ShellCheck runs before adding new
     ambient providers or reviewed divergences.
