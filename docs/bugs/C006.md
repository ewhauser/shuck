## Delta analysis

### Shell/runtime variables and preseeded environment names `[shuck-fix]`

**Bucket:** Shuck-only
**Impact:** Most of the sample

Shuck is flagging names that the shell or process environment already provides, even when the script never assigns them itself. The repeated examples include Bash state such as `LINENO`, `FUNCNAME`, `BASH_SOURCE`, `BASH_LINENO`, `BASH_REMATCH`, `RANDOM`, `READLINE_LINE`, `BASH_VERSION`, and `BASH_VERSINFO`, plus common environment names such as `USER`, `HOME`, `SHELL`, `PWD`, `TERM`, `LANG`, `OSTYPE`, `HISTCONTROL`, and `HISTSIZE`.

ShellCheck stays silent on these cases, so this looks like a missing exemption in C006 rather than a semantic disagreement.

**Samples:**
- `89luca89__distrobox__distrobox-ephemeral:35` - `SUDO_USER` and `DOAS_USER` are read from the login environment.
- `89luca89__distrobox__distrobox-ephemeral:43` - `USER`, `HOME`, and `SHELL` are normalized from the current account.
- `Bash-it__bash-it__completion__available__aliases.completion.bash:27` - `RANDOM` feeds `mktemp`.
- `Bash-it__bash-it__completion__available__aliases.completion.bash:31` - `BASH_REMATCH` comes from bash pattern matching.
- `Bash-it__bash-it__plugins__available__history.plugin.bash:12` - `HISTCONTROL` is part of the shell's own history state.
- `Bash-it__bash-it__plugins__available__percol.plugin.bash:30` - `READLINE_LINE` is a readline-provided buffer.
- `Bash-it__bash-it__themes__clean__clean.theme.bash:5` - theme variables such as `red` and `white` are injected by the theme framework.
- `Bash-it__bash-it__plugins__available__docker.plugin.bash:59` - `OSTYPE` is a shell-provided platform hint.

### Generated helpers and long scope chains `[shuck-fix]`

**Bucket:** Shuck-only
**Impact:** Large, but concentrated in a smaller family of scripts

A second pattern shows up in helper-heavy scripts where values are clearly established earlier in the same file, or are threaded through generated helper bodies, but C006 still reports them as never initialized. The affected names include `distrobox_path`, `container_manager`, `container_additional_packages`, `ROOTDIR`, `BOOT`, `TARGET`, `params`, `__backtitle`, `md_inst`, `configdir`, `my_alias`, `CRASHDIR`, `as_lineno`, and `ac_var`-style configure cache variables.

ShellCheck does not flag these locations, which points to a scope/project-closure accounting gap in shuck rather than an intentional difference.

**Samples:**
- `89luca89__distrobox__distrobox-ephemeral:108` - `distrobox_path` is used later in the same script to dispatch helper commands.
- `89luca89__distrobox__distrobox-rm:350` - `container_manager` is reused across the command-building path.
- `void-linux__void-packages__srcpkgs__prometheus__files__prometheus__run:6` - `MAX_OPEN_FILES`, `WRKDIR`, and `ARGS` are part of the package's startup contract.
- `void-linux__void-packages__common__build-style__gemspec.sh:21` - `pkgname` and `version` are template inputs, not missing assignments.
- `juewuy__ShellCrash__scripts__libs__set_profile.sh:5` - `my_alias`, `shtype`, and `CRASHDIR` are threaded into generated profile content.
- `ko1nksm__shellspec__libexec__shellspec-translate.sh:22` - `specfile` and `lineno` are part of the translator's own state.
- `moovweb__gvm__examples__native__configure:1868` - the long trap body references configure-script cache variables that are populated by the generated flow.
- `moovweb__gvm__examples__native__configure:2052` - `ac_var` and its related cache state are internal configure bookkeeping.

### Small ShellCheck-only tail in helper-heavy scripts `[shuck-fix]`

**Bucket:** ShellCheck-only
**Impact:** 11 locations / 10 scripts

There is a small opposite tail where ShellCheck still reports a few undefined-variable hits and shuck misses them. These are all in long helper-heavy or generated scripts, not in a clean standalone repro. The affected files cluster around RetroPie module scripts, ShellCrash helper setup, shellspec translation, the generated `configure` script, and the gemspec build helper.

This looks like the same scope/project-closure area as the larger shuck-only family, just in the opposite direction.

**Samples:**
- `RetroPie__RetroPie-Setup__scriptmodules__supplementary__audiosettings.sh:62` - helper-local menu state inside a function.
- `RetroPie__RetroPie-Setup__scriptmodules__supplementary__wifi.sh:110` - dialog menu state threaded through a helper path.
- `RetroPie__RetroPie-Setup__scriptmodules__supplementary__skyscraper.sh:171` - generated menu selection inside a nested helper.
- `juewuy__ShellCrash__scripts__libs__set_profile.sh:5` - profile rewriting code that relies on surrounding shell state.
- `ko1nksm__shellspec__libexec__shellspec-translate.sh:22` - translator state used while emitting generated shell.
- `moovweb__gvm__examples__native__configure:1868` - the generated trap body is the biggest ShellCheck-only outlier.
- `void-linux__void-packages__common__build-style__gemspec.sh:22` - templated gemspec rewrites still have one ShellCheck-only hit in the sample.

## Notes

This rule is not close to parity yet. The sample shows a broad over-reporting problem, with most of the work likely in the scope and special-variable model rather than in narrow per-script exceptions.

## 2026-04-08 update

Re-ran `make test-large-corpus SHUCK_LARGE_CORPUS_RULES=C006` on April 8, 2026 after teaching `C006` to ignore unresolved environment-style names that are entirely uppercase, numeric, or underscore-delimited. That one rule-local filter cut the shuck-only side from 37,230 locations in the full targeted run to 9,830, while the ShellCheck-only side stayed at 53.

The remaining gap is no longer dominated by plain environment names. The largest surviving clusters are helper-heavy or project-local flows such as `neofetch`, `rvm`, `acme.sh`, `void-packages`, `termux-packages`, and `pi-hole`, with labels concentrated in `directive-handling`, `project-closure`, `shell-collapse`, and helper-library/test-harness contexts. That points the next round toward source-closure and generated-helper modeling rather than another simple name filter in `undefined_variable.rs`.

Later the same day, three more fixes brought the full targeted run down again:

- Guarded parameter expansions such as `${name:-fallback}`, `${name:=value}`, `${name:+alt}`, and `${name:?message}` no longer feed `C006`, and `${name:=value}` now seeds later reads in the semantic model. That moved the `shuck-only` side from `9,830` to `8,335`.
- `C006` now follows ShellCheck's broader SC2154 policy for script-local names by suppressing reports when a non-function binding for that name exists anywhere in the file, even if the binding is later, conditional, or tucked inside another helper. That cut the `shuck-only` side from `8,335` to `2,401`.
- Associative-array literal keys now stay out of the undefined-variable path, so cases like `map[swift-cmark]=1` and `${map[swift-cmark]}` no longer invent reads of `swift` and `cmark`. That trimmed the `shuck-only` side again from `2,401` to `2,276`.

The ShellCheck-only side stayed flat at `53` through all three follow-up fixes. The remaining shuck-only tail is now concentrated in unlabeled helper-library globals such as the `rvm` fetch/install helpers plus the existing `project-closure` and `shell-collapse` buckets. The next likely win is better modeling for globals that arrive from sourced project state or shared helper initialization, not more per-name filtering.

A fourth follow-up narrowed the remaining guard-related false positives without reopening the opposite side. `C006` now suppresses names that are only used in plain truthiness and string-unary presence guards such as `[ "$name" ]`, `[ -z "$name" ]`, `[ -n "$name" ]`, `[[ "$name" ]]`, and logical combinations of those forms in `[[ ... ]]`. The collector intentionally skips file tests and binary comparisons like `[[ -s "$path" ]]` or `[[ "$name" = x ]]`, since ShellCheck still reports those. That brought the full targeted run from `2,276` down to `1,916` shuck-only locations while the overall ShellCheck-only total stayed at `53`.

This pass cleared most of the acme.sh deploy/config families (`DirectAdmin_*`, `Netlify_*`, `PEPLINK_*`, `Le_DeployHook`) and a smaller set of helper-local guard cases, but the remaining tail is still dominated by `project-closure` and `shell-collapse` buckets:

- `project-closure`: `672`
- `(none)`: `671`
- `shell-collapse`: `331`
- `project-closure,test-harness`: `66`
- `project-closure,shell-collapse`: `58`

The biggest remaining scripts are now `powerlevel10k`'s `gitstatus` build helper plus the same `rvm` and `void-packages` scope-heavy families (`manage__base_install`, `manage__base_fetch`, `pkg`, `mount`, `void-cross.sh`, `99-pkglint.sh`, `pycompile`). That keeps the next likely win in the scope/source-closure area rather than in more name-shape filtering.
