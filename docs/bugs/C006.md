## Delta analysis

### Ambient shell and package variables `[shuck-fix]`

**Bucket:** Shuck-only  
**Impact:** Dominates the sample

The biggest remaining gap is that shuck reports a very large number of variables that ShellCheck treats as ambient, framework-provided, or otherwise already-known in these scripts. The sample is full of package/build variables and shell/runtime names such as `VERSION`, `BUILD`, `OUTPUT`, `PKGTYPE`, `TAG`, `ARCH`, `ROOTDIR`, `TERMUX_PREFIX`, `BITNAMI_COLOR`, `PWD`, `HOME`, `USER`, `SHELL`, `OSTYPE`, `RANDOM`, `BASH_REMATCH`, `READLINE_LINE`, `COMP_WORDS`, `XDG_RUNTIME_DIR`, and `LANG`.

Representative scripts:

- `void-linux__void-packages__srcpkgs__prometheus__files__prometheus__run`
- `void-linux__void-packages__srcpkgs__m1n1__files__kernel.d__m1n1.post-install`
- `Bash-it__bash-it__completion__available__grunt.completion.bash`
- `Bash-it__bash-it__plugins__available__percol.plugin.bash`
- `CISOfy__lynis__include__tests_boot_services`

The current rule only knows about file-local assignment state, so anything coming from an ambient shell environment or package framework looks like a typo. That is too broad for SC2154 parity.

### Directive-handling and project-closure scripts `[environment]`

**Bucket:** Shuck-only  
**Impact:** 390 directive-handling blocks, 556 project-closure blocks in the sample

Many of the remaining false positives sit in scripts where ShellCheck is already quiet because of inline `# shellcheck disable=SC2154` directives or because the file participates in a larger sourced/helper context. Shuck still emits diagnostics there, which makes the sample look much noisier than ShellCheck.

Representative scripts:

- `Bash-it__bash-it__completion__available__aliases.completion.bash`
- `Bash-it__bash-it__completion__available__travis.completion.bash`
- `HariSekhon__DevOps-Bash-tools__.bash.d__mac.sh`
- `HariSekhon__DevOps-Bash-tools__aws__aws_csv_creds.sh`
- `HariSekhon__DevOps-Bash-tools__aws__aws_secret_add_binary.sh`

This looks like a combination of suppression handling and file-closure modeling, not a core undefined-variable message bug.

### Definite misses in straight-line scripts `[shuck-fix]`

**Bucket:** ShellCheck-only  
**Impact:** 11 locations across 9 scripts

The shellcheck-only tail is small, but it is real. In the sample, ShellCheck reports a few undefined reads that shuck misses entirely, mostly in scripts that use assignment/condition chains or helper logic that our current dataflow does not recover well enough.

Representative script:

- `233boy__v2ray__src__log.sh`

Representative missed names from that file:

- `is_core`
- `is_log_dir`
- `is_config_json`

These look like true under-reports rather than span-only mismatches.
