## Delta Analysis

### Captured Substitutions With Internal Redirect Plumbing `[shuck-fix]`

**Bucket:** Shuck-only
**Impact:** 165 locations / 31 scripts

The dominant pattern is command substitutions whose result is still being consumed, but which contain internal redirections or fd-swapping plumbing. Shuck treats any stdout redirection inside the substitution as a reason to warn, which catches interactive wrappers, `getopt`/`jq`/`awk` helpers, and `dialog`/`whiptail` menu commands even when the substitution's output is intentionally captured.

Representative samples:
- `HariSekhon__DevOps-Bash-tools__cicd__selenium_hub_wait_ready.sh:76` - `ready="$(jq -r '.value.ready' <<< "$status" || die "FAILED to parse Selenium Hub response: $status" >&2)"`
- `docker-library__official-images__test__run.sh:22` - `opts="$(getopt ... -- "$@" || { usage >&2 && false; })"`
- `community-scripts__ProxmoxVE__tools__pve__fstrim.sh:75` - `excluded_containers_raw=$(whiptail ... 3>&1 1>&2 2>&3)`
- `RetroPie__RetroPie-Setup__scriptmodules__supplementary__gamecondriver.sh:177` - `local choice=$("${cmd[@]}" "${options[@]}" 2>&1 >/dev/tty)`
- `rvm__rvm__scripts__gemsets:378:16-381:4` - command substitution in a project-closure helper context

Several of the same shapes appear in `project-closure` and `directive-handling` scripts, but they are still the same shuck-only substitution warning rather than a separate semantic class.
