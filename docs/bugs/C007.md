## Delta Analysis

### `find | xargs` span anchor drifts `[location-only]`

**Bucket:** Location-only
**Impact:** 161 locations / 143 scripts

This is the dominant mismatch. Shuck and ShellCheck agree that the `find ... | xargs ...` pattern is worth warning about, but Shuck usually anchors the span across the whole pipeline or continuation line while ShellCheck points at the `find` invocation or the immediate unsafe handoff. That makes the diagnostics look different even when they describe the same underlying issue.

**Samples:**
- `SlackBuildsOrg__slackbuilds__academic__getfem__getfem.SlackBuild:105` - `find $PKG -iname "*.so" | xargs chmod +x`
- `SlackBuildsOrg__slackbuilds__desktop__redshift__redshift.SlackBuild:99` - `find $PKG | xargs file | grep -e "executable" -e "shared object" | grep ELF | cut -f 1 -d : | xargs strip --strip-unneeded`
- `SlackBuildsOrg__slackbuilds__development__gprbuild__gprbuild.SlackBuild:114-115` - two adjacent `find -name ... | xargs rm` lines with split spans
- `SlackBuildsOrg__slackbuilds__development__notepadnext__notepadnext.SlackBuild:106-109` - multiline `find ... | xargs rm -f`

### `find | xargs -I` helper copy pipelines `[shuck-correct]`

**Bucket:** Shuck-only
**Impact:** 3 locations / 1 script

ShellCheck stays quiet on the Bitnami Jenkins helper-copy pipelines, but the command still moves file names through `xargs` without a null-delimited boundary. Because those names can contain whitespace, I think Shuck is right to warn here even though ShellCheck does not.

**Samples:**
- `bitnami__containers__bitnami__jenkins__2__debian-12__rootfs__opt__bitnami__scripts__libjenkins.sh:216` - mounted-content copy pipeline
- `bitnami__containers__bitnami__jenkins__2__debian-12__rootfs__opt__bitnami__scripts__libjenkins.sh:426` - plugin copy pipeline
- `bitnami__containers__bitnami__jenkins__2__debian-12__rootfs__opt__bitnami__scripts__libjenkins.sh:479` - nested mounted-path copy pipeline

### Parser aborts and source-closure noise `[environment]`

**Bucket:** Environment
**Impact:** 11 parse-abort fixtures and 2 project-closure fixtures

These cases are not useful rule-level parity signals. Some fixtures stop in Shuck's parser before the lint comparison can run, and a small tail depends on sourced-helper resolution or local directive behavior. I excluded them from the bucket math because they are not direct `C007` implementation disagreements.

**Samples:**
- `GameServerManagers__LinuxGSM__lgsm__modules__alert_ifttt.sh` - parser abort on an unterminated command substitution
- `GameServerManagers__LinuxGSM__lgsm__modules__alert_telegram.sh` - parser abort on the same shape
- `SlackBuildsOrg__slackbuilds__development__aflplusplus__mkqemutarball.sh:24` - `find . -name .git\* | xargs rm -rf`
- `bitnami__containers__bitnami__jenkins__2__debian-12__rootfs__opt__bitnami__scripts__libjenkins.sh:216` - project-closure/directive-handling tail around helper-file execution
