## Delta Analysis

### Declaration assignments outside `export` are being missed `[shuck-fix]`

**Bucket:** ShellCheck-only  
**Impact:** 345 locations / 64 scripts in the sample

The sampled misses are dominated by ordinary declaration assignments that ShellCheck treats as SC2155 but S010 does not currently match. The current rule only looks for `export` declarations with command substitutions, so it stays silent on `local`, `declare`, and similar assignment-bearing declarations even when they capture command output directly.

Representative examples:
- `RetroPie__RetroPie-Setup__scriptmodules__emulators__flycast.sh:80` - `local temp_conf="$(mktemp)"` is flagged by ShellCheck, but S010 ignores it.
- `termux__termux-packages__packages__svt-av1__build.sh:19` - `local current_soversion=$(sed ...)` shows the same pattern in a build helper.
- `bittorf__kalua__openwrt-addons__etc__kalua__db:1122` - `local border="$( ... )"` is another direct command-output capture in a declaration.
- `community-scripts__ProxmoxVE__misc__build.func:1840,1844` - more SC2155 hits on plain assignments inside helper code.

### `export foo=$(...)` is detected, but the span is too wide `[location-only]`

**Bucket:** Location-only  
**Impact:** 29 scripts in the sample

When S010 does fire on an `export` assignment, it tends to anchor the diagnostic on the whole assignment word or the full right-hand side. ShellCheck's SC2155 span is narrower and usually starts at the command substitution or the value being captured. That creates a consistent location-only drift rather than a semantic disagreement.

Representative examples:
- `SlackBuildsOrg__slackbuilds__desktop__happy-hacking-gnu__happy-hacking-gnu.SlackBuild:43` - `export ARCH=$( uname -m )`
- `SlackBuildsOrg__slackbuilds__development__Bear__Bear.SlackBuild:93` - `export CARGO_HOME=$(pwd)/.cargo`
- `SlackBuildsOrg__slackbuilds__misc__txt2tags__txt2tags.SlackBuild:63` - `export PYTHONPATH=$PKG/$(python3 -c ...)`
- `SlackBuildsOrg__slackbuilds__libraries__libpgf__libpgf.SlackBuild:39` - `export ARCH=$(...)` with the wider Shuck span
- `void-linux__void-packages__common__xbps-src__libexec__xbps-src-dopkg.sh:40` - `export XBPS_ARCH=$(<$XBPS_MASTERDIR/.xbps_chroot_init)`

### Helper and closure-heavy scripts still produce noisy edge cases `[environment]`

**Bucket:** Environment  
**Impact:** 11 scripts in the sample

These fixtures never reached a clean rule comparison because parsing stopped first. They are useful for corpus health tracking, but they do not point to an `S010` matcher bug by themselves.

**Samples:**
- `GameServerManagers__LinuxGSM__lgsm__modules__alert_ifttt.sh` - unterminated command substitution
- `HariSekhon__DevOps-Bash-tools__git__git_diff_commit.sh` - unterminated double quote
- `SlackBuildsOrg__slackbuilds__games__njam__patches__drop_gda.patch` - parser expected a word

## Checklist

- [ ] Broaden `export_command_substitution.rs` so declaration assignments with command substitutions are matched outside `export`.
  - The sampled misses are mostly `local`, `declare`, and similar declaration forms that ShellCheck treats as SC2155.
  - Add regressions for `local temp_conf="$(mktemp)"`, `local current_soversion=$(...)`, and `local border="$(...)"` shapes.
- [ ] Tighten the span selection for the `export foo=$(...)` cases.
  - The current report lands too far to the right compared with ShellCheck's narrower SC2155 span.
  - Recheck the `ARCH`, `CARGO_HOME`, `PYTHONPATH`, and `XBPS_ARCH` examples after the anchoring fix.
- [ ] Re-run the 10% corpus sample after the matcher and span adjustments.

## Notes

Update on 2026-04-06: `S010` now shares command-substitution detection with the common classifier layer, but it is still intentionally limited to `export` declarations in this phase.

This sample did not suggest an intentional divergence. The main gap is that S010 is still export-only, while ShellCheck's SC2155 behavior covers broader declaration assignments that capture command output directly.
