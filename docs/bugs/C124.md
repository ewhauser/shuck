## Delta analysis

### Guarded exits are over-reported `[shuck-fix]`

**Bucket:** Shuck-only
**Impact:** Most of the sample

The repeated shape is a guard that exits on one path, but still falls through on the other path. Shuck is treating the code after that guard as unreachable even though the script can still reach it. ShellCheck stays quiet on these same locations, so this looks like an over-eager reachability analysis rather than a span issue.

The common forms include `return 1 || exit 1`, `command || exit 1`, and `if ...; then exit 1; fi` style guards where later code is still reachable when the condition fails.

**Samples:**
- `HariSekhon__DevOps-Bash-tools__checks__check_no_tabs.sh:52` - `return 1 &>/dev/null || exit 1`
- `HariSekhon__DevOps-Bash-tools__checks__check_readme_badges.sh:71` - a guarded failure path inside a helper function
- `HariSekhon__DevOps-Bash-tools__setup__ci_bootstrap.sh:41` - a guarded failure path after setup steps
- `SlackBuildsOrg__slackbuilds__development__quickjs__quickjs.SlackBuild:82` - `patch -p1 < ... || exit 1;`
- `SlackBuildsOrg__slackbuilds__network__pCloudDrive__pCloudDrive.SlackBuild:48` - a large guarded block that still has live fallthrough paths
- `acmesh-official__acme.sh__notify__smtp.sh:111` - several branches exit on failures, but later code is still reachable on success
- `termux__termux-packages__x11-packages__libncnn__build.sh:74` - a guarded build block that should not make later lines unreachable
- `void-linux__void-packages__common__hooks__post-install__01-remove-misc.sh:8` - a post-install helper with a guarded exit path
