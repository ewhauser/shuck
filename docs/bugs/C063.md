## Delta analysis

### ShellSpec helper factories are over-flagged `[shuck-fix]`

**Bucket:** Shuck-only
**Impact:** 272 locations / 8 scripts

Most of the sample is the same pattern repeated across ShellSpec helper files: a wrapper function defines a helper function body, calls it immediately, and then later wrappers intentionally reuse the same helper name again. ShellCheck stays quiet, but Shuck reports the earlier definitions as overwritten before they can be called. In these files the repeated definitions are part of the DSL or test harness, not a real shadowing bug.

**Samples:**
- `ko1nksm__shellspec__lib__core__matchers__be__variable.sh:11` - `shellspec_matcher__match` is redefined inside each matcher wrapper
- `ko1nksm__shellspec__libexec__shellspec-list.sh:67` - `specfile` is intentionally rebound across helper blocks
- `ko1nksm__shellspec__libexec__shellspec-translate.sh:251` - `progress` is defined in the translator helper and reused by the surrounding shellspec code
- `ko1nksm__shellspec__spec__core__clone_spec.sh:101` - repeated `var1` and `vars` helper definitions across generated spec cases
- `ko1nksm__shellspec__spec__core__dsl_spec.sh:1013` - repeated `prepare`, `mock`, `block`, and `shellspec_output` definitions across the DSL fixture
- `ko1nksm__shellspec__spec__core__evaluation_spec.sh:12` - repeated `evaluation_execute`, `prepare`, and `evaluation` helpers
- `ko1nksm__shellspec__spec__core__matchers__equal_spec.sh:15` - repeated `subject` helpers
- `ko1nksm__shellspec__spec__core__modifiers__result_spec.sh:40` - repeated `subject`, `mock`, and `success_with_output` helpers

### Mocked helper swaps in test harnesses are over-flagged `[shuck-fix]`

**Bucket:** Shuck-only
**Impact:** 12 locations / 5 scripts

The `nvm` and `rvm` fixtures deliberately `unset -f` and redefine helper functions to simulate different environments and return values during a single test file. ShellCheck does not warn on these rebinding patterns, but Shuck does. These are test doubles, not accidental overwrites.

**Samples:**
- `nvm-sh__nvm__test__fast__Unit tests__nvm_compare_checksum:15` - `nvm_compute_checksum` is redefined between assertions
- `nvm-sh__nvm__test__fast__Unit tests__nvm_curl_libz_support:12` - `curl` is swapped between libz and non-libz test cases
- `nvm-sh__nvm__test__fast__Unit tests__nvm_ensure_version_installed:46` - `nvm_has_system_node` and `nvm_has_system_iojs` are rebound to model each branch
- `nvm-sh__nvm__test__fast__Unit tests__nvm_remote_version:14` - `nvm_ls_remote`, `nvm_ls_remote_iojs`, and `nvm_remote_versions` are intentionally replaced in later checks
- `rvm__rvm__scripts__notes:16` - `new_notes` is defined differently per branch before `eval "${PAGER:-\command \cat}"`

### Repeated helper definitions in ordinary shell scripts are over-flagged `[shuck-fix]`

**Bucket:** Shuck-only
**Impact:** 5 locations / 4 scripts

A smaller tail appears in normal shell helpers where a function is defined, used, and then intentionally defined again later in the same file or generated script. These look like generated helper bodies or branch-local overrides, not real call-site bugs.

**Samples:**
- `bitnami__containers__bitnami__wordpress-nginx__6__debian-12__rootfs__opt__bitnami__scripts__libphp.sh:127` - `php_execute_print_output` is duplicated later in the file
- `bitnami__containers__bitnami__wordpress-nginx__6__debian-12__rootfs__opt__bitnami__scripts__libphp.sh:146` - `php_execute` is also duplicated
- `bittorf__kalua__openwrt-addons__etc__kalua__tunnel:357` - `list_files` is defined separately inside case branches
- `community-scripts__ProxmoxVE__misc__build.func:1430` - `_is_whitelisted_key` is overwritten in the generated helper flow
- `gentoo__gentoo__app-shells__bash__files__bashrc.d__10-gentoo-title-r3.bash:6` - `genfun_set_win_title` is redefined inside itself as part of the title helper pattern
