## Delta Analysis

### ShellSpec harnesses trigger most of the sampled over-reporting `[shuck-fix]`

**Bucket:** Shuck-only  
**Impact:** 272 locations / 8 scripts in the sample

The largest cluster is in `ko1nksm/shellspec` files, where the same helper names are intentionally redefined many times across generated harness code and nested matcher helpers. Shuck reports each redefinition as an overwritten function even when the files are clearly framework machinery, while ShellCheck stays silent for the sample.

The repeated names are things like `shellspec_matcher__match`, `subject`, `evaluation`, `prepare`, `block`, `mock`, and `shellspec_output`. That pattern suggests the current call-graph-based rule is too eager for generated test DSL code and harness-heavy files.

**Samples:**
- `ko1nksm__shellspec__lib__core__matchers__be__variable.sh:11` - repeated `shellspec_matcher__match` definitions inside harness helpers
- `ko1nksm__shellspec__spec__core__clone_spec.sh:101` - dozens of repeated helper redefinitions in one spec file
- `ko1nksm__shellspec__spec__core__dsl_spec.sh:1013` - large generated spec file with repeated helper names
- `ko1nksm__shellspec__spec__core__evaluation_spec.sh:12` - nested evaluation helpers repeatedly redefine the same names
- `ko1nksm__shellspec__spec__core__modifiers__result_spec.sh:9` - small but similar harness-driven overwrite site

### NVM test helpers also over-report `[shuck-fix]`

**Bucket:** Shuck-only  
**Impact:** 10 locations / 4 scripts in the sample

The sampled `nvm` hits are test-harness functions that reuse the same helper names across different cases. They look more like fixture scaffolding than user-facing function overwrite bugs, and ShellCheck remains silent on them in the sampled corpus.

**Samples:**
- `nvm-sh__nvm__test__fast__Unit tests__nvm_compare_checksum:15` - `nvm_compute_checksum` is defined twice in the same test fixture
- `nvm-sh__nvm__test__fast__Unit tests__nvm_curl_libz_support:12` - `curl` helper reuse inside the test harness
- `nvm-sh__nvm__test__fast__Unit tests__nvm_ensure_version_installed:46` - repeated `nvm_has_system_node` / `nvm_has_system_iojs` helpers
- `nvm-sh__nvm__test__fast__Unit tests__nvm_remote_version:14` - repeated remote-version helpers

### Smaller helper and project files follow the same shape `[shuck-fix]`

**Bucket:** Shuck-only  
**Impact:** 7 locations / 5 scripts in the sample

The remaining hits are still function-overwrite reports, but they are concentrated in helper scripts, sourced project files, or generated utility code. ShellCheck does not flag them in the sample, which again points to `C063` being too broad for framework-style or repeated helper definitions.

**Samples:**
- `bitnami__containers__bitnami__wordpress-nginx__6__debian-12__rootfs__opt__bitnami__scripts__libphp.sh:127` - duplicate `php_execute_print_output` / `php_execute` definitions
- `bittorf__kalua__openwrt-addons__etc__kalua__tunnel:357` - repeated `list_files` helper definition
- `community-scripts__ProxmoxVE__misc__build.func:1430` - repeated `_is_whitelisted_key` helper in a project-closure file
- `gentoo__gentoo__app-shells__bash__files__bashrc.d__10-gentoo-title-r3.bash:6` - `genfun_set_win_title` overwritten in a shell init fragment
- `rvm__rvm__scripts__notes:16` - repeated `new_notes` helper in sourced notes logic

## Checklist

- [ ] Decide whether `overwritten_function.rs` should ignore generated harnesses and DSL-heavy test files such as ShellSpec.
  - The shellspec cluster is the main source of sampled over-reporting.
  - If these files should stay in scope, add targeted regressions and a policy note; otherwise add a guard or allowlist.
- [ ] Decide whether repeated helper definitions in test fixtures like `nvm` should be treated as intentional framework code rather than `C063` bugs.
  - If intentional, document the divergence and add the reviewed corpus entries to the appropriate allowlist.
- [ ] Re-run the 10% corpus sample after the matcher or policy change.

## Notes

This sample had no ShellCheck-only or location-only mismatches. The entire gap is shuck over-reporting, and most of the sampled locations came from ShellSpec harnesses, so the first improvement should focus on generated framework code rather than general call-graph logic.
