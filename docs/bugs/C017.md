## Delta analysis

### Unary literal tests are under-reported `[shuck-fix]`

**Bucket:** ShellCheck-only  
**Impact:** 3 locations / 3 scripts in the sample

The current `C017` matcher handles simple binary comparisons, but it does not cover unary literal tests such as `-n` on a fixed string. ShellCheck reports those as constant checks, while shuck stays silent in the sampled corpus.

**Samples:**
- `nvm-sh__nvm__test__fast__Unit tests__nvm_get_colors:9` - `if [ -n TEMP_NVM_COLORS ]; then`
- `nvm-sh__nvm__test__fast__Unit tests__nvm set_colors:10` - `if [ -n TEMP_NVM_COLORS ]; then`
- `nvm-sh__nvm__test__fast__Set Colors__nvm_print_versions calls nvm_get_colors:14` - `if [ -n TEMP_NVM_COLORS ]; then`

### Constant-looking tests that still depend on runtime state are over-reported `[shuck-fix]`

**Bucket:** Shuck-only  
**Impact:** 2 locations / 2 scripts in the sample

Two sampled hits show that the current matcher is too eager about anything that parses as a fixed word sequence. In one case, a bare identifier inside `[[ ... -ge ... ]]` is actually a loop variable, not a fixed literal. In the other, a file-order comparison like `-ot` depends on filesystem state even when the path names themselves are static.

**Samples:**
- `RetroPie__RetroPie-Setup__scriptmodules__supplementary__skyscraper.sh:685` - `[[ i -ge 10 ]]` inside a `for i in ...` loop; `i` is a runtime variable, not a fixed literal
- `bittorf__kalua__openwrt-addons__usr__sbin__cron.webcam:338` - `[ "/webcam/lastsnap.jpg" -ot "/webcam/MARK" ]` depends on file metadata, not just literal text

### Parse-abort fixtures are corpus noise `[environment]`

**Bucket:** Environment

These failures never reached rule comparison because shuck rejected the script first. They are useful for corpus health tracking, but they are not actionable `C017` parity work by themselves.
