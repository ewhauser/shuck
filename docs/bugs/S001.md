## Delta Analysis

### Missing unquoted-expansion hits in ordinary command and assignment contexts `[shuck-fix]`

**Bucket:** ShellCheck-only  
**Impact:** 30,929 locations / 118 scripts in the sample

This is the main parity gap. ShellCheck continues to flag many plain unquoted expansions in argument position, assignment values, and command substitutions where Shuck stays silent. The sampled scripts show the rule still missing straightforward cases like exported environment values, path concatenation, and `${...}` expansions that reach executable arguments unchanged.

The examples also suggest the misses are not limited to one syntax form. Some appear in `export NAME=$VALUE` style assignments, some in command arguments, and some inside larger command substitutions that should still inherit the underlying unquoted-expansion warning.

**Samples:**
- `void-linux__void-packages__srcpkgs__apache-kafka__files__apache-kafka.sh:3` - ShellCheck flags the unquoted `PATH` append in an `export` assignment; Shuck does not.
- `void-linux__void-packages__srcpkgs__Gokapi__files__gokapi__run:12` - ShellCheck sees the unquoted user and group values in the `chpst -u` invocation; Shuck only catches part of the expression.
- `void-linux__void-packages__srcpkgs__netsurf__files__netsurf.sh:2-3` - the `echo $LANG` pipeline and launcher wrapper still produce ShellCheck-visible unquoted-expansion hits that Shuck does not fully match.

### Span anchoring is still drifting on matched hits `[location-only]`

**Bucket:** Location-only  
**Impact:** 113 locations / 113 scripts in the sample

These fixtures agree on the diagnostic code, but the spans do not line up. That makes this bucket lower priority than true miss/overreport cases, but it is still worth tracking because it shows the rule is not anchoring exactly where ShellCheck does.

The drift shows up in small shell fragments where the expansion is embedded in filenames or option values. The underlying semantics are aligned, but Shuck often marks a wider or narrower slice than ShellCheck.

**Samples:**
- `void-linux__void-packages__srcpkgs__booster__files__kernel-hook-postrm:10` - both tools warn on the `VERSION` expansion in the filename, but the spans differ.
- `void-linux__void-packages__srcpkgs__tinyramfs__files__kernel-hook-postinst:13` - ShellCheck and Shuck agree on the risky expansion, yet disagree on the exact extent of the affected text.
- `void-linux__void-packages__srcpkgs__dracut__files__kernel-uefi-hook-postinst:20` - the bundle-path and kernel-cmdline expansions are semantically the same problem, but the highlighted range is not identical.

### Shuck is reporting extra hits in packaging and bootstrap scripts `[shuck-fix]`

**Bucket:** Shuck-only  
**Impact:** 19,228 locations / 378 scripts in the sample

The largest over-reporting cluster is in generated or packaging-style shell scripts. These are mostly ordinary shell files, but they tend to contain environment bootstrap code, `export` setup, and command wrappers where Shuck is currently stricter than ShellCheck.

In the sampled fixtures, Shuck is especially eager around environment setup lines and wrapper commands. The problem is not isolated to one repository or one coding style; it shows up in package scripts, container entrypoints, and other maintenance-oriented shell code.

**Samples:**
- `void-linux__void-packages__srcpkgs__Gokapi__files__gokapi__run:6-9` - Shuck flags several default-value assignments that ShellCheck leaves alone in this bootstrap context.
- `void-linux__void-packages__srcpkgs__dracut__files__kernel-uefi-hook-postinst:19` - Shuck reports the bundle directory expansion inside the `mkdir` setup line where ShellCheck stays quiet.
- `void-linux__void-packages__srcpkgs__netsurf__files__netsurf.sh:8` - Shuck adds an extra `exit $?`-adjacent hit that does not appear in ShellCheck’s output for the sample.

### Directive handling and source-resolution noise should be separated before deeper parity work `[environment]`

**Bucket:** Parse-abort / environment  
**Impact:** 387 labeled occurrences across 266 scripts

This bucket is a mix of directive-handling, project-closure, and unknown-shell-collapse cases. They are important to track, but they are not clean rule-local parity misses. The corpus shows multiple fixtures where the tool disagreement is driven by inline suppression directives, sourced-file resolution, or shell-detection ambiguity rather than by the unquoted-expansion logic itself.

These should not be used as primary evidence for changing `unquoted_expansion.rs` until the surrounding environment behavior is understood or excluded.

**Samples:**
- `89luca89__distrobox__distrobox-ephemeral` - contains an inline `shellcheck disable` directive and several environment-setup blocks that make the diff harder to attribute to S001 alone.
- `89luca89__distrobox__distrobox-rm` - similar directive-heavy bootstrap logic with multiple sourced config paths.
- `Bash-it__bash-it__vendor__github.com__rcaloras__bash-preexec__bash-preexec.sh` - mixes directive handling with a shell-detection ambiguity caused by the file header.
- `void-linux__void-packages__srcpkgs__dracut__files__kernel-uefi-hook-postinst:19-20` - also appears in the main diff buckets, but the broader project-closure behavior means it should be rechecked after the environment cases are separated out.

## Checklist

- [ ] Tighten the span capture in `unquoted_expansion.rs` so ordinary argument and assignment expansions line up more closely with ShellCheck.
  - Focus first on argument lists, export-style assignments, and command substitutions that currently lose part of the warning or miss it entirely.
- [ ] Review the environment/bootstrap cluster separately from the core rule matcher.
  - Decide whether directive-handling, project-closure, and unknown-shell-collapse fixtures should be excluded from S001 parity work or handled with dedicated allowlist coverage.
- [ ] Add focused regressions for the representative packaging and bootstrap scripts from the sample.
  - The goal is to lock in both the true misses and the span-anchoring behavior once the matcher is adjusted.
- [ ] Re-run the large-corpus comparison after the matcher and environment-policy decisions are made.

## Notes

This report does not add allowlist entries yet. The current priority is to reduce the large body of ShellCheck-only misses, then clean up the smaller location-only drift once the core matcher is closer to parity.
