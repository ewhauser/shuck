## Delta Analysis

### Dynamic Parameter-Expansion Patterns `[shuck-correct]`

**Bucket:** Shuck-only  
**Impact:** 152 locations / 33 scripts

Every sampled compatibility diff is the same broad shape: shuck warns when a variable is expanded inside a parameter-expansion pattern, while ShellCheck stays silent. That includes prefix/suffix stripping, replacement, and nested pattern rewrites such as `${source_path/%$source_name/$destination_name}` and `${CC/${target}-/}`. The rule is behaving consistently, and the sampled cases look like useful diagnostics rather than accidental noise.

**Samples:**
- `Bash-it__bash-it__completion__available__aliases.completion.bash:95,126` - dynamic pattern pieces inside shell-completion helpers
- `Bash-it__bash-it__vendor__github.com__rcaloras__bash-preexec__bash-preexec.sh:117,118,326,327,328` - multiple variable-driven pattern expansions in the same helper
- `HariSekhon__DevOps-Bash-tools__azure_devops__azure_devops_api.sh:134` - variable expansion inside a pattern expression
- `rvm__rvm__scripts__gemsets:121,137,138,139,388,576,579,584,630` - repeated prefix/suffix replacements built from runtime state
- `community-scripts__ProxmoxVE__misc__build.func:1072,1073,1096,1458,1459` - several dynamic patterns in project-local helper code
- `termux__termux-packages__packages__ghc__build.sh:107,108` - nested replacement patterns for toolchain prefixes
- `void-linux__void-packages__common__build-style__gemspec.sh:167` - dynamic pattern replacement in a packaging helper
