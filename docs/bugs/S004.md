## Delta Analysis

### Span anchoring is drifting on otherwise matched command-substitution hits `[location-only]`

**Bucket:** Location-only  
**Impact:** 358 locations / 292 scripts in the sample

This is the dominant sample pattern. Both tools agree that the script should warn, but Shuck often highlights a slightly wider span than ShellCheck. The drift shows up most often in ordinary command arguments where `$(...)` is embedded in a command line or build recipe and the warning boundary lands a few characters differently.

The pattern looks systematic rather than random. In the sampled fixtures, Shuck usually captures a wider slice of the substitution or adjacent text, while ShellCheck keeps the warning focused more tightly on the substitution itself.

**Samples:**
- `SlackBuildsOrg__slackbuilds__academic__artha__artha.SlackBuild:83` - both tools warn on the `$(readlink $i)` / `$(...)` area, but the exact span differs by a few characters.
- `SlackBuildsOrg__slackbuilds__academic__grpn__grpn.SlackBuild:76` - the command substitution in the build command is the same semantic issue, but Shuck and ShellCheck disagree on the boundary.
- `SlackBuildsOrg__slackbuilds__academic__harminv__harminv.SlackBuild:127` - another ordinary command-argument hit with span drift only.

### Nested substitutions in assignments and compound commands are still being missed `[shuck-fix]`

**Bucket:** ShellCheck-only  
**Impact:** 62 locations / 47 scripts in the sample

This is the main semantic gap. ShellCheck is still finding unquoted command substitutions inside `local` assignments, `[[ ... ]]` tests, here-strings, and wrapped command invocations where Shuck stays silent. That points to the matcher being too narrow, not just a span-anchoring issue.

The sampled scripts suggest the rule is strongest in plain argument position, but it still misses nested or contextual forms of the same problem. In practice that means the current implementation needs to descend into more expression shapes, not just top-level command arguments.

**Samples:**
- `233boy__v2ray__src__log.sh:24` - `cat <<<$(jq ...)` is flagged by ShellCheck, but Shuck misses it.
- `233boy__v2ray__src__log.sh:27` - another `cat <<<$(jq ...)` form with the same under-reporting.
- `termux__termux-packages__scripts__build__termux_step_get_dependencies_python.sh:18-19` - `local name_python_module=$(sed ... | awk ...)` and the follow-on `local` assignment both still trigger ShellCheck-only hits.
- `RetroPie__RetroPie-Setup__scriptmodules__supplementary__runcommand__runcommand.sh:1309` - the nested `$(ps ... $(ps ... $PPID))` test is still visible to ShellCheck but not to Shuck.

### The Shuck-only cluster mixes one real false positive with directive-handling noise `[shuck-fix]`

**Bucket:** Shuck-only  
**Impact:** 10 locations / 6 scripts in the sample

There are two subpatterns here. The clear false positive is a quoted substitution that ShellCheck leaves alone, which means Shuck is treating a safe form as unsafe. The rest of the cluster is mostly suppression or closure-related noise, where ShellCheck directives or project-closure semantics change what gets reported and Shuck is not matching that behavior yet.

The actionable part is to separate these concerns before changing the core matcher further. The quoted-substitution case is a direct rule bug; the directive-heavy and project-closure fixtures need suppression-aware handling so they do not pollute the core parity signal.

**Samples:**
- `thoughtbot__dotfiles__bin__git-ca:3` - `--date="$(date ...)"` is already quoted, but Shuck still reports `SC2046`.
- `Bash-it__bash-it__plugins__available__docker.plugin.bash:40` - `# shellcheck disable=SC2046` immediately precedes the unquoted substitution, but Shuck still emits `S004`.
- `Bash-it__bash-it__plugins__available__docker.plugin.bash:49` - another suppression-protected `docker rmi $(...)` site that ShellCheck suppresses.
- `HariSekhon__DevOps-Bash-tools__.bash.d__mac.sh:45` - `# shellcheck disable=SC2046` is present before `eval $(/opt/homebrew/bin/brew shellenv)`, but Shuck still flags it.
- `HariSekhon__DevOps-Bash-tools__lib__gcp_ci.sh:142` - same suppression-aware mismatch in a project-closure fixture.

## Checklist

- [ ] Broaden `unquoted_command_substitution.rs` so it descends into nested substitutions in `[[ ... ]]`, `local name=$(...)`, here-strings, and wrapped invocations like `bash -c "... $(...) ..."`.
  - The ShellCheck-only cluster is mostly made of those shapes.
- [ ] Tighten span capture for the ordinary command-argument cases that already match semantically.
  - The location-only bucket is large, but it looks like span anchoring drift rather than a logic bug.
- [ ] Verify suppression handling for `# shellcheck disable=SC2046` and confirm whether the quoted-substitution hit in `thoughtbot__dotfiles__bin__git-ca` is enough to justify a dedicated regression.
  - This separates true over-reporting from directive-handling noise.
- [ ] Re-run the 10% corpus sample after the matcher and suppression-path follow-up.
