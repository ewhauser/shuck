## Delta Analysis

### Sentinel literal `-n` tests in `nvm` harnesses are over-reported `[shuck-fix]`

**Bucket:** Shuck-only  
**Impact:** 3 locations / 3 scripts in the sample

All sampled `C019` mismatches came from `nvm`'s own test harnesses, where the unary `-n` check is used against a fixed sentinel string rather than against user input or runtime state. Shuck currently reports these as fixed-literal string tests, but ShellCheck leaves them alone. The common shape is a setup or cleanup guard like `if [ -n TEMP_NVM_COLORS ]; then ...; fi`, where the literal token is part of the test fixture state rather than a bug-prone hard-coded comparison.

The current matcher in `literal_unary_string_test.rs` is therefore too broad for this corpus slice: it treats the literal sentinel check as a problem even though ShellCheck does not.

**Samples:**
- `nvm-sh__nvm__test__fast__Set Colors__nvm_print_versions calls nvm_get_colors:14` - `if [ -n TEMP_NVM_COLORS ]; then`
- `nvm-sh__nvm__test__fast__Unit tests__nvm set_colors:10` - `if [ -n TEMP_NVM_COLORS ]; then`
- `nvm-sh__nvm__test__fast__Unit tests__nvm_get_colors:9` - `if [ -n TEMP_NVM_COLORS ]; then`
