# 010: Zsh Dialect Support

## Status

Proposed

## Summary

Add parser-level zsh dialect support to the shuck workspace and thread that dialect through `shuck`, `shuck-linter`, and `shuck-formatter`. The design keeps a single AST crate by treating `shuck-ast` as a superset shell AST factored by syntax family rather than by language. Shared syntax families continue to use common nodes. Where Bash and zsh genuinely diverge, the AST introduces exhaustive family-specific enums so consumers are forced to decide how to handle each form. Only constructs with no honest cross-shell counterpart become standalone variants.

The initial milestone is parse-and-preserve support for real zsh source, not full runtime execution compatibility. Full zsh runtime semantics, option-sensitive emulation modes, and complete formatter coverage are explicitly out of scope for the first implementation.

## Motivation

The repository already partially knows about zsh:

- File discovery treats `.zsh` files and `#!/usr/bin/env zsh` shebangs as shell scripts.
- `shuck-linter` has a `ShellDialect::Zsh` value and can infer it from path or shebang.
- Portability rules already reserve codes for zsh-only syntax.

But the parser and formatter do not yet have first-class zsh support:

- `shuck-parser::ShellDialect` only distinguishes `Posix`, `Mksh`, and `Bash`.
- `shuck check` parses with `Parser::new(...)`, which defaults to Bash behavior.
- The formatter dialect enum and config parsing do not accept `zsh`.
- Real zsh syntax such as `${(M)${(k)parameters[@]}:#pattern}` and `${(%):-%x}` does not fit the current Bash-shaped parameter expansion model.

This leaves an awkward gap:

- A file can be identified as zsh but still fail to parse as Bash.
- Parser-level failures block later linting and formatting work.
- Future portability rules for zsh-only syntax have no structured AST to inspect.

Supporting zsh at the lexer/parser/AST level gives the project a path to:

- Parse real zsh startup scripts, completions, and plugin code without falling back to Bash guesswork.
- Preserve exact spans and surface forms for later linting and formatting.
- Reuse one AST and one parser pipeline across shell dialects instead of creating a parallel zsh frontend.

## Design

### Goals

- Add `ShellDialect::Zsh` to `shuck-parser` and propagate that dialect through CLI, config, linter, and formatter entry points.
- Parse common zsh source with exact span preservation and without degrading existing Bash, POSIX, or mksh behavior.
- Keep a single shared AST crate.
- Represent zsh-only syntax structurally enough that later rules and formatting logic can reason about it.
- Make parser consumers that do not yet understand zsh-specific semantics behave conservatively rather than incorrectly.

### Non-Goals

- Implement the zsh execution/runtime model in `shuck-parser`.
- Support every zsh option-dependent grammar mode such as `emulate`, `KSH_ARRAYS`, or `SH_WORD_SPLIT`.
- Fully model completion-system DSLs, full glob-qualifier semantics, or prompt expansion in the first milestone.
- Guarantee pretty-printing for every zsh-only node in the formatter before parser support lands.
- Fully generalize every future shell dialect beyond Bourne-family and zsh as part of the first milestone.

### Architecture Overview

The dialect flow after this change:

```
Source text + path/shebang/config
    -> resolved shell dialect
    -> Lexer + Parser with dialect feature table
    -> Script AST + comments
    -> Linter / Formatter / CLI reporting
```

The key design choice is that dialect is an input to parsing and analysis, not a property baked into every AST node. The AST remains syntax-oriented and dialect-neutral where possible. Surface distinctions that matter for round-tripping or portability checks are stored on the nodes that need them.

`Parser::new(input)` remains Bash by default for backward compatibility in low-level tests and existing callers. Workspace-level entry points should prefer `Parser::with_dialect(input, dialect)` after resolving dialect from shebang, extension, CLI flags, or config.

### Dialect Plumbing

`shuck-parser` grows a zsh parser dialect:

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash, Default)]
pub enum ShellDialect {
    Posix,
    Mksh,
    #[default]
    Bash,
    Zsh,
}
```

The workspace then resolves and threads the dialect consistently:

- `shuck check` must parse with the inferred shell dialect instead of always calling `Parser::new(...)`.
- `shuck-formatter::ShellDialect` gains `Zsh`.
- `[format].dialect = "zsh"` and `--dialect zsh` become valid formatter inputs.
- Any parser-adjacent helpers that currently collapse `zsh` into Bash defaults must be updated to preserve `Zsh`.

This is intentionally end-to-end even though the parser work is the core of the feature. A parser dialect that cannot be selected from the CLI or formatter is not operationally useful.

### Single AST Model

The AST remains shared across dialects, but the "best design" version of that idea is stricter than simply bolting on zsh-specific nodes.

The model uses four rules:

1. Factor the tree by syntax family, not by language.
2. Use exhaustive enums at the smallest meaningful point of divergence.
3. Preserve exact surface form on the family node that owns the divergence.
4. Add standalone variants only for constructs with no real shared family.

This yields one superset AST rather than parallel Bash and zsh trees, while still forcing conscious handling where the grammars truly differ.

#### Shared Syntax Reuses Existing Nodes

These constructs already have workable AST shapes and should remain shared:

| Syntax family | Existing AST shape |
|---|---|
| `[[ ... ]]` conditionals | `CompoundCommand::Conditional` |
| `(( ... ))` arithmetic commands | `CompoundCommand::Arithmetic` |
| `for ((...))` arithmetic loops | `CompoundCommand::ArithmeticFor` |
| `function name` forms | `FunctionDef` + `FunctionSurface` |
| Process substitution `<(...)` / `>(...)` | `WordPart::ProcessSubstitution` |
| Arrays, subscripts, and selectors | `VarRef`, `Subscript`, `ArrayAccess`, `ArraySlice`, etc. |

Where zsh accepts the same surface form with no extra structural requirements, the parser should enable the existing parser branch in zsh mode rather than creating a duplicate node kind.

#### Syntax Enums at Divergence Points

Some zsh syntax is semantically close to an existing construct but uses a different surface grammar. These should stay inside the shared family node, but the divergence should be explicit and exhaustively matchable.

For example, zsh brace-style `if` should still parse as `IfCommand`, but the `if` node should carry a required `IfSyntax` enum rather than ad hoc optional spans:

```rust
pub struct IfCommand {
    pub condition: Vec<Command>,
    pub then_branch: Vec<Command>,
    pub elif_branches: Vec<(Vec<Command>, Vec<Command>)>,
    pub else_branch: Option<Vec<Command>>,
    pub syntax: IfSyntax,
    pub span: Span,
}

pub enum IfSyntax {
    ThenFi {
        then_span: Span,
        fi_span: Span,
    },
    Brace {
        left_brace_span: Span,
        right_brace_span: Span,
    },
}
```

The AST consumer that cares about source form can match `IfSyntax`. The consumer that only cares about control flow can ignore it.

This rule generalizes:

- `FunctionDef` should continue to use a nested syntax/surface type as it already does.
- Command-list background forms should be modeled by an operator enum, not by a zsh-only command node.
- Redirect operators, case terminators, and similar punctuation families should follow the same pattern if new dialect-specific forms are added later.

#### Standalone Variants for Shell-Only Constructs

Some constructs do not map cleanly onto the current Bash-oriented model and need dedicated nodes:

```rust
pub enum CompoundCommand {
    If(IfCommand),
    For(ForCommand),
    ArithmeticFor(Box<ArithmeticForCommand>),
    While(WhileCommand),
    Until(UntilCommand),
    Case(CaseCommand),
    Select(SelectCommand),
    Subshell(Vec<Command>),
    BraceGroup(Vec<Command>),
    Arithmetic(ArithmeticCommand),
    Time(TimeCommand),
    Conditional(ConditionalCommand),
    Coproc(CoprocCommand),
    Always(AlwaysCommand),
}

pub struct AlwaysCommand {
    pub body: Vec<Command>,
    pub always_body: Vec<Command>,
    pub span: Span,
}
```

The `Always` node is intentionally additive. It is not representable as a normal `IfCommand`, `BraceGroup`, or `CommandList` without losing the cleanup/finally-style relationship between the two bodies.

### Parser Feature Model

The current parser mostly gates syntax through helpers like `ensure_bash_or_mksh()` and `ensure_bash_only()`. That is too coarse once zsh is added. The parser should move to a dialect feature table:

```rust
pub struct DialectFeatures {
    pub double_bracket: bool,
    pub arithmetic_command: bool,
    pub arithmetic_for: bool,
    pub function_keyword: bool,
    pub select_loop: bool,
    pub coproc_keyword: bool,
    pub zsh_parameter_modifiers: bool,
    pub zsh_brace_if: bool,
    pub zsh_always: bool,
    pub zsh_background_operators: bool,
}
```

Each parser entry point consults this table instead of hard-coding "Bash only" or "Bash or mksh". This keeps feature admission explicit and makes zsh additions incremental instead of invasive.

For the initial implementation, the feature families are:

| Feature family | Strategy |
|---|---|
| Shared Bash-like syntax accepted by zsh | Reuse the existing parser branch and enable it through `DialectFeatures`. |
| Parameter expansion families | Parse through one unified parameter-expansion family with exhaustive Bourne vs zsh sub-enums. |
| Brace-form `if [[ ... ]] { ... }` | Parse as `IfCommand` with `IfSyntax`. |
| `always` blocks | Add `CompoundCommand::Always`. |
| Zsh background/operator forms such as `&|` and `&!` | Extend command-list surface representation. |
| Zsh glob controls and qualifier suffixes such as `(#i)*.jpg`, `(#b)(*)`, `*(.)`, and `*.log(#qN)` | Parse one dedicated glob word-part node with ordered pattern/control segments plus one optional terminal qualifier group, and keep analysis conservative. |
| Option-dependent and niche syntax | Defer to later phases. |

### Command List Surface for Zsh Operators

The current `ListOperator` conflates all background-like forms into `Background`. Zsh needs distinct surface forms preserved:

```rust
pub enum ListOperator {
    And,
    Or,
    Semicolon,
    Background(BackgroundOperator),
}

pub enum BackgroundOperator {
    Plain,   // &
    Pipe,    // &|
    Bang,    // &!
}
```

Corresponding lexer tokens should be introduced for the zsh-only operators so the parser does not have to reconstruct them from raw text. This is a surface-model change, not a runtime-model commitment: the AST only needs to preserve which operator appeared and where.

### Parameter Expansion Family

Parameter expansion is the largest parser/AST gap and the place where exhaustive dialect branching is most useful.

The current AST spreads parameter-related syntax across many `WordPart` variants:

- `Variable`
- `ParameterExpansion`
- `Length`
- `ArrayAccess`
- `ArrayLength`
- `ArrayIndices`
- `Substring`
- `ArraySlice`
- `IndirectExpansion`
- `PrefixMatch`
- `Transformation`

That structure is already Bash-shaped. The best long-term design is to replace that spread with one parameter-expansion family node and put Bash/zsh divergence inside that family.

The new family should represent:

- Leading modifier groups like `${(M)...}` or `${(k)...}`
- Modifier groups with arguments and custom separators
- Expansions with no normal variable head, such as `${(%):-%x}`
- Modifier-bearing nested expansions that still use Bash-like tails

The parameter family should look like:

```rust
pub enum WordPart {
    Literal(LiteralText),
    SingleQuoted { value: SourceText, dollar: bool },
    DoubleQuoted { parts: Vec<WordPartNode>, dollar: bool },
    Parameter(ParameterExpansion),
    CommandSubstitution { commands: Vec<Command>, syntax: CommandSubstitutionSyntax },
    ArithmeticExpansion { expression: SourceText, expression_ast: Option<ArithmeticExprNode>, syntax: ArithmeticExpansionSyntax },
    ProcessSubstitution { commands: Vec<Command>, is_input: bool },
}

pub struct ParameterExpansion {
    pub syntax: ParameterExpansionSyntax,
    pub span: Span,
    pub raw_body: SourceText,
}

pub enum ParameterExpansionSyntax {
    Bourne(BourneParameterExpansion),
    Zsh(ZshParameterExpansion),
}

pub enum BourneParameterExpansion {
    Simple { reference: VarRef },
    Length { reference: VarRef },
    Access { reference: VarRef },
    Indirect {
        name: Name,
        operator: Option<ParameterOp>,
        operand: Option<SourceText>,
        colon_variant: bool,
    },
    PrefixMatch {
        prefix: Name,
        kind: PrefixMatchKind,
    },
    Slice {
        reference: VarRef,
        offset: SourceText,
        offset_ast: Option<ArithmeticExprNode>,
        length: Option<SourceText>,
        length_ast: Option<ArithmeticExprNode>,
    },
    Operation {
        reference: VarRef,
        operator: ParameterOp,
        operand: Option<SourceText>,
        colon_variant: bool,
    },
    Transformation {
        reference: VarRef,
        operator: char,
    },
}

pub struct ZshParameterExpansion {
    pub target: ZshExpansionTarget,
    pub modifiers: Vec<ZshModifier>,
    pub operation: Option<ZshExpansionOperation>,
}

pub enum ZshExpansionTarget {
    Reference(VarRef),
    Nested(Box<ParameterExpansion>),
    Empty,
}

pub struct ZshModifier {
    pub name: char,
    pub argument: Option<SourceText>,
    pub argument_delimiter: Option<char>,
    pub span: Span,
}

pub enum ZshExpansionOperation {
    PatternOperation {
        kind: ZshPatternOp,
        operand: SourceText,
    },
    Defaulting {
        kind: ZshDefaultingOp,
        operand: SourceText,
        colon_variant: bool,
    },
    Raw(SourceText),
}
```

This design is intentionally family-first:

- every parameter expansion is found in one place in the tree
- matching on `ParameterExpansionSyntax` forces a conscious Bourne vs zsh decision
- `WordPart` no longer has to carry a growing pile of Bash-only special cases
- nested zsh modifier forms can still recurse into the same family
- `raw_body` preserves the full contents between `${` and `}` for verbatim formatting and future lowering.

If effort is truly free, this is better than simply adding `WordPart::ZshExpansion`, because it puts the dialect split at the precise family boundary where consumers need to reason about it.

### Conservative Consumer Semantics

The initial zsh support is parser-first. That means other layers must handle new nodes conservatively before they become fully semantic-aware.

#### Linter

Until rule logic learns more about zsh semantics:

- `WordPart::Parameter(_)` remains the one expansion-bearing entry point for parameter syntax.
- `ParameterExpansionSyntax::Zsh(_)` is treated as an expansion-bearing dynamic form.
- Safe-value and field-splitting analyses should assume "unknown / potentially unsafe" rather than claiming a literal result.
- Rules that only care about presence of expansion continue to work.
- Rules that require Bourne-specific parameter semantics should skip or conservatively downgrade when they encounter `ParameterExpansionSyntax::Zsh(_)`.

This is preferable to misclassifying zsh expansions using Bourne rules.

#### Formatter

Formatter support is staged:

- `format_source()` and `format_script_ast()` must accept `zsh` as an input dialect.
- For nodes without dedicated zsh formatting logic, the formatter may emit the original source slice via verbatim fallback.
- Simplify rewrites must leave zsh-only nodes untouched until they are explicitly taught how to preserve zsh semantics.

This allows zsh files to enter the formatter pipeline immediately without requiring every printer rule to be rewritten first.

### Phased Rollout

#### Phase 1: Dialect Selection and Shared Syntax

- Add `ShellDialect::Zsh` to `shuck-parser`.
- Thread zsh through CLI/config/formatter/linter call sites.
- Replace coarse parser gates with `DialectFeatures`.
- Enable shared existing AST nodes where zsh uses an already-supported surface form.

#### Phase 2: AST Refactor and Zsh Grammar Additions

- Replace the spread parameter-related `WordPart` variants with a unified `ParameterExpansion` family.
- Add `ParameterExpansionSyntax::{Bourne, Zsh}`.
- Add `IfSyntax` to preserve brace-style `if`.
- Add `CompoundCommand::Always`.
- Extend command-list operators for zsh background/operator forms.

#### Phase 3: Conservative Consumer Integration

- Teach linter common helpers to classify `ParameterExpansionSyntax::Zsh(_)` conservatively.
- Teach formatter to accept zsh nodes and fall back to verbatim slices where needed.
- Add targeted portability rules and AST queries once the syntax is structurally available.

#### Phase 4: Follow-On Semantics

Future work, intentionally outside the first milestone:

- Option-sensitive parsing and analysis for `emulate` and shell options.
- More typed zsh operation variants instead of `Raw(SourceText)`.
- Dedicated formatter rules for zsh-only syntax.
- Additional zsh glob grammar beyond the supported parser subset, such as other `(#...)` option letters, quoted cases, embedded ambiguous groups, or other niche forms.

## Alternatives Considered

### Alternative A: Separate Zsh AST

Build a second AST family for zsh and translate only shared nodes between the two worlds.

Rejected because this duplicates almost every consumer boundary: query helpers, span logic, formatter rules, linter walkers, and test fixtures would all need parallel zsh-aware entry points. The project already has a strong shared-AST architecture; zsh should extend it rather than bypass it.

### Alternative B: Keep the AST Split by Language at High Levels

Introduce `BashWordPart`, `ZshWordPart`, `BashCommand`, and similar language-specific trees or wrappers across broad portions of the AST.

Rejected because it makes most consumers branch in places where the syntax family is actually shared. Pipelines, simple commands, and most compound constructs should remain common nodes. Broad language-splitting is less composable than putting exhaustive enums at the specific divergence point.

### Alternative C: Treat Zsh-Only Forms as Opaque Literal Words

Teach the lexer to keep zsh syntax together as `Word` tokens, but do not add structured AST nodes for zsh-only constructs.

Rejected because it postpones the hard part without creating a useful platform. Lint rules could not inspect the syntax, formatters could only echo raw text, and span-level diagnostics for zsh-only constructs would still be difficult. If the parser is going to accept the syntax, it should preserve structure where the syntax boundary is already known.

### Alternative D: Fully Generalize All Parameter Expansions Before Adding Zsh

Keep the current spread Bash parameter variants and bolt on an extra `WordPart::ZshExpansion` node beside them.

Rejected under the "effort is free" design goal because it puts the dialect split in the wrong place. Parameter expansion is one syntax family. The AST should force exhaustive handling at that family boundary instead of scattering Bash forms across many variants and hanging zsh off the side as a special case.

## Security Considerations

Zsh parameter syntax introduces more nested delimiters, modifier groups, and raw subregions. The parser must not respond to that by adding unbounded backtracking or recursive descent with uncontrolled depth.

The same protections already used by `shuck-parser` must continue to apply:

- Parser fuel limits still cap total work.
- Command substitution and nested `${...}` parsing still obey depth limits.
- Modifier parsing must be linear in the input length of the expansion body.
- `Raw(SourceText)` stores source slices only; it does not imply evaluation or interpretation.

Formatter verbatim fallback is also part of the safety story: if a zsh-only node is parsed but not yet structurally printable, the formatter should emit the original source slice rather than attempting a lossy rewrite.

## Verification

- **Dialect plumbing**: `.zsh` files and `#!/usr/bin/env zsh` files are parsed with `ShellDialect::Zsh` in `shuck check` and `shuck format`.
- **Shared syntax enablement**: existing node families such as `[[ ... ]]`, arithmetic commands, function keyword definitions, and process substitutions parse successfully in zsh mode without AST regressions in Bash, mksh, or POSIX mode.
- **Zsh parameter parsing**: targeted unit tests verify that `${(M)${(k)parameters[@]}:#__gitcomp_builtin_*}`, `${(%):-%x}`, and `${(m)foo}` parse into `ParameterExpansionSyntax::Zsh(_)` with preserved modifier order and raw body spans.
- **Parameter family exhaustiveness**: parameter-expansion consumers match on `ParameterExpansionSyntax::{Bourne, Zsh}` rather than relying on ad hoc zsh detection in `WordPart`.
- **Surface preservation**: brace-form `if` records `IfSyntax`, `always` blocks produce `CompoundCommand::Always`, and zsh background/operator forms preserve their exact operator kind in `ListOperator`.
- **Loop preservation**: `repeat` and `foreach` parse into dedicated compound-command nodes with preserved header words, body spans, and syntax-family enums for brace vs `do`/`done` spellings.
- **Conservative analysis**: linter common helpers do not classify `ParameterExpansionSyntax::Zsh(_)` as a fixed literal or otherwise-safe value by default.
- **Formatter safety**: formatting a zsh file in zsh mode is parse-stable, and unsupported zsh-only nodes round-trip through verbatim fallback without source corruption.
- **Dialect gating**: Bash, POSIX, and mksh modes continue rejecting the zsh-only `repeat` and `foreach` loop spellings.
- **Real-world fixtures**: parser tests cover the existing `zsh-idioms.test.sh` cases and add focused zsh fixtures for nested modifier groups, brace-form conditionals, `always` blocks, and `repeat` / `foreach` loops.

```bash
# Parser unit and integration tests
cargo test -p shuck-parser

# Formatter tests, including zsh dialect resolution and verbatim fallback
cargo test -p shuck-formatter

# CLI/config integration for dialect plumbing
cargo test -p shuck -- check
cargo test -p shuck -- format

# Workspace smoke test
make test
```

The zsh feature is complete for its first milestone when a `.zsh` file that uses common zsh parameter modifiers and narrow loop forms such as `repeat` or `foreach` parses successfully through `shuck check`, produces a structured AST, and can be passed through the formatter in zsh mode without corruption.
