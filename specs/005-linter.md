# 005: Linter

## Status

Proposed

## Summary

A new `shuck-linter` library crate that implements rule dispatch and diagnostic reporting for shell script linting. Rules are organized into categories with prefix-based codes and dispatched through a hybrid model: inline traversal hooks share the semantic builder for node-local and evaluation-order-sensitive checks, while post-traversal phases query the finalized semantic model for whole-file facts. The design is modeled after ruff's `ruff_linter` architecture, adapted for shell-specific concerns and shell semantics.

Suppressions and auto-fixes are explicitly out of scope. Output formatters are out of scope.

## Motivation

Today `shuck check` only reports parse errors. The Go frontend defines 332 lint rules across semantic, dataflow, and project phases. To ship any of these rules, we need:

- **A rule registry** — A way to define rules with codes, categories, severity levels, and metadata. Rules must be selectable by prefix (`C`, `C0`, `C001`).
- **A dispatch mechanism** — A way to invoke rules against analyzed files. Most rules should query semantic entities (bindings, references, scopes, call sites) after the walk finishes, but some checks need evaluation-order hook points while the semantic traversal is happening.
- **Structured diagnostics** — A diagnostic type that carries severity, rule code, message, and source location. The CLI can format these for output; the linter produces them.

Ruff solves this for Python with a `Checker` that visits AST nodes while building semantics, then runs follow-on analyses over the collected model. We adapt that pattern to shell: the semantic traversal exposes internal observer hooks for inline checks, while the public rule-authoring model remains semantic-first and query-based for most rules.

## Design

### Rule Code Scheme

Rules use a single-letter category prefix followed by a three-digit number, following ruff's naming convention (e.g., `E101`, `F401`, `B006`):

| Prefix | Category | Description | Example |
|--------|----------|-------------|---------|
| `C` | Correctness | Bugs, undefined variables, unreachable code, incorrect usage | `C001` |
| `S` | Style | Quoting, formatting, naming conventions, idiomatic patterns | `S001` |
| `P` | Performance | Unnecessary subprocesses, useless subshells, inefficient patterns | `P001` |
| `X` | Portability | POSIX compliance, bashisms, cross-shell compatibility | `X001` |
| `K` | Security | Injection risks, unsafe eval, unvalidated input | `K001` |

**Prefix-based selection** works by string prefix matching on the full code:

- `C` → all correctness rules
- `C0` → correctness rules `C001`–`C099`
- `C001` → single rule
- `ALL` → all rules across all categories

This gives each category 999 rule slots — more than enough. The scheme is extensible: new categories get new prefixes without renumbering.

**Mapping from Go rules:** The 332 existing Go rules (currently numbered `SH-001` through `SH-351` with gaps) will be remapped into the new category scheme. The old `SH-NNN` codes become aliases for backward compatibility with existing `# shuck: disable=SH-001` directives in user scripts.

### Core Types

#### Rule Registry

The `Rule` enum is generated by a declarative macro that maps codes to variants and derives metadata:

```rust
/// Declares all rules with their code, category, default severity, and violation type.
macro_rules! declare_rules {
    ($(
        ($code:literal, $category:path, $severity:path, $name:ident),
    )*) => {
        #[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
        #[repr(u16)]
        pub enum Rule {
            $($name,)*
        }

        impl Rule {
            pub const fn code(&self) -> &'static str {
                match self {
                    $(Rule::$name => $code,)*
                }
            }

            pub const fn category(&self) -> Category {
                match self {
                    $(Rule::$name => $category,)*
                }
            }

            pub const fn default_severity(&self) -> Severity {
                match self {
                    $(Rule::$name => $severity,)*
                }
            }
        }

        /// Look up a Rule from its code string.
        pub fn code_to_rule(code: &str) -> Option<Rule> {
            match code {
                $($code => Some(Rule::$name),)*
                _ => None,
            }
        }
    };
}

declare_rules! {
    // Correctness
    ("C001", Category::Correctness, Severity::Error,   UndefinedVariable),
    ("C002", Category::Correctness, Severity::Error,   UnusedAssignment),
    ("C003", Category::Correctness, Severity::Error,   UnreachableCode),
    // ...

    // Style
    ("S001", Category::Style, Severity::Warning, UnquotedExpansion),
    // ...

    // Performance
    ("P001", Category::Performance, Severity::Warning, UselessCat),
    // ...

    // Portability
    ("X001", Category::Portability, Severity::Warning, Bashism),
    // ...

    // Security
    ("K001", Category::Security, Severity::Error,   UnsafeEval),
    // ...
}
```

#### RuleSet

An efficient bitset for O(1) rule membership checks, following ruff's `RuleSet`:

```rust
/// Bitset over the Rule enum. Supports up to 1024 rules.
pub struct RuleSet([u64; 16]);

impl RuleSet {
    pub const EMPTY: Self = Self([0; 16]);

    pub const fn contains(&self, rule: Rule) -> bool {
        let idx = (rule as u16) / 64;
        let bit = (rule as u16) % 64;
        (self.0[idx as usize] & (1 << bit)) != 0
    }

    pub fn insert(&mut self, rule: Rule) { /* set bit */ }
    pub fn remove(&mut self, rule: Rule) { /* clear bit */ }
    pub fn union(&self, other: &Self) -> Self { /* bitwise OR */ }
    pub fn subtract(&self, other: &Self) -> Self { /* bitwise AND NOT */ }
    pub fn iter(&self) -> impl Iterator<Item = Rule> { /* yield set bits */ }
    pub fn len(&self) -> usize { /* popcount */ }
    pub fn is_empty(&self) -> bool { self.0.iter().all(|&w| w == 0) }
}
```

#### RuleSelector

Parses user-facing rule selection strings into sets of rules:

```rust
pub enum RuleSelector {
    /// All rules: "ALL"
    All,
    /// All rules in a category: "C", "S", etc.
    Category(Category),
    /// Rules matching a code prefix: "C0" → C001..C099
    Prefix(String),
    /// Single rule: "C001"
    Rule(Rule),
}

impl FromStr for RuleSelector {
    type Err = SelectorParseError;

    fn from_str(s: &str) -> Result<Self, Self::Err> {
        if s == "ALL" {
            return Ok(Self::All);
        }
        // Try exact rule match first
        if let Some(rule) = code_to_rule(s) {
            return Ok(Self::Rule(rule));
        }
        // Try category match
        if let Some(cat) = match s {
            "C" => Some(Category::Correctness),
            "S" => Some(Category::Style),
            "P" => Some(Category::Performance),
            "X" => Some(Category::Portability),
            "K" => Some(Category::Security),
            _ => None,
        } {
            return Ok(Self::Category(cat));
        }
        // Try prefix match — verify at least one rule matches
        if Rule::iter().any(|r| r.code().starts_with(s)) {
            return Ok(Self::Prefix(s.to_string()));
        }
        Err(SelectorParseError::Unknown(s.to_string()))
    }
}

impl RuleSelector {
    /// Expand this selector into the set of matching rules.
    pub fn into_rule_set(&self) -> RuleSet {
        match self {
            Self::All => { /* all rules */ }
            Self::Category(cat) => { /* rules where rule.category() == cat */ }
            Self::Prefix(p) => { /* rules where rule.code().starts_with(p) */ }
            Self::Rule(r) => { /* singleton */ }
        }
    }
}
```

### Violations and Diagnostics

#### Violation Trait

The trait is intentionally minimal. No fix-related methods — those will be added when fixes are in scope.

pub struct UndefinedVariable {

}

impl Violation for UndefinedVariable {
    fn rule() -> Rule { Rule::UndefinedVariable }

    fn message(&self) -> String {
        format!("undefined variable `{}`", self.name)

}

pub fn undefined_variable(checker: &mut Checker) {
    for &ref_id in checker.semantic().unresolved() {
        let reference = checker.semantic().reference(ref_id);

        // Skip known shell built-in variables ($?, $#, $@, etc.)
        if is_builtin_variable(&reference.name) {
            continue;
        }

        checker.report(
            UndefinedVariable { name: reference.name.clone() },
            reference.span,
        );
    }
}

### Checker and Dispatch

The `Checker` is the central orchestrator. Unlike the original query-only design, the updated architecture is **hybrid**: an internal traversal observer handles inline checks during semantic construction, and the post-traversal `Checker` iterates semantic entities for rules that need the completed model.

```rust
pub struct Checker<'a> {
    /// The semantic model for this file.
    semantic: &'a SemanticModel,
    /// The positional index for this file.
    indexer: &'a Indexer,
    /// The parsed AST (available for structural rules).
    script: &'a Script,
    /// Source text.
    source: &'a str,
    /// Enabled rules.
    rules: &'a RuleSet,
    /// Collected diagnostics.
    diagnostics: Vec<Diagnostic>,
}

impl<'a> Checker<'a> {
    pub fn semantic(&self) -> &'a SemanticModel { self.semantic }
    pub fn indexer(&self) -> &'a Indexer { self.indexer }
    pub fn ast(&self) -> &'a Script { self.script }
    pub fn source(&self) -> &'a str { self.source }

    #[inline]
    pub fn is_rule_enabled(&self, rule: Rule) -> bool {
        self.rules.contains(rule)
    }

    pub fn report<V: Violation>(&mut self, violation: V, span: Span) {
        self.diagnostics.push(Diagnostic::new(violation, span));
    }

    /// Run all enabled rules and return collected diagnostics.
    pub fn check(mut self) -> Vec<Diagnostic> {
        self.check_bindings();
        self.check_references();
        self.check_scopes();
        self.check_declarations();
        self.check_call_sites();
        self.check_source_refs();
        self.check_commands();
        self.check_flow();
        self.diagnostics
    }
}
```

#### Dispatch Phases

Rules are dispatched in phases based on what semantic entity they query. Each phase iterates the relevant entities and calls enabled rules:

```rust
impl<'a> Checker<'a> {
    /// Phase 1: Rules that check variable/function bindings.
    /// Covers: unused assignments, overwritten functions, readonly reassignment.
    fn check_bindings(&mut self) {
        for binding in self.semantic.bindings() {
            if self.is_rule_enabled(Rule::UnusedAssignment) {
                correctness::unused_assignment::unused_assignment(self, binding);
            }
            // ...more binding rules
        }
    }

    /// Phase 2: Rules that check variable/function references.
    /// Covers: undefined variables, unquoted expansions.
    fn check_references(&mut self) {
        if self.is_rule_enabled(Rule::UndefinedVariable) {
            correctness::undefined_variable::undefined_variable(self);
        }
        // ...more reference rules
    }

    /// Phase 3: Rules that check scopes.
    /// Covers: function-level scope issues, local-outside-function.
    fn check_scopes(&mut self) {
        for scope in self.semantic.scopes() {
            if self.is_rule_enabled(Rule::LocalOutsideFunction) {
                // ...
            }
        }
    }

    /// Phase 4: Rules that check declaration builtins.
    /// Covers: declare misuse, redundant export, conflicting attributes.
    fn check_declarations(&mut self) {
        for decl in self.semantic.declarations() {
            // ...
        }
    }

    /// Phase 5: Rules that check function call sites.
    /// Covers: calling undefined functions, wrong argument count.
    fn check_call_sites(&mut self) {
        for (name, sites) in self.semantic.call_sites() {
            // ...
        }
    }

    /// Phase 6: Rules that check source/import references.
    /// Covers: dynamic source paths, missing sourced files.
    fn check_source_refs(&mut self) {
        for source_ref in self.semantic.source_refs() {
            // ...
        }
    }

    /// Phase 7: Structural AST rules that need direct tree access.
    /// Covers: useless cat, pipeline patterns, command-specific checks.
    /// This is the only phase that walks the AST.
    fn check_commands(&mut self) {
        if self.is_rule_enabled(Rule::UselessCat) {
            performance::useless_cat::useless_cat(self);
        }
        // ...more structural rules
    }

    /// Phase 8: Dataflow rules (require CFG + reaching definitions).
    /// Covers: dead code, variable overwrite before read.
    fn check_flow(&mut self) {
        if !self.rules_need_dataflow() {
            return; // Skip building CFG if no dataflow rules are enabled
        }
        // ...
    }
}
```

The phase ordering is intentional: inline traversal hooks run during semantic construction, while semantic entity iteration (phases 1–6) handles the majority of rules after the model is complete. Structural rules (phase 7) and dataflow rules (phase 8) still run last and only when needed.

### Settings

```rust
pub struct LinterSettings {
    /// Enabled rules (resolved from user selectors).
    pub rules: RuleSet,
    /// Per-rule severity overrides.
    pub severity_overrides: FxHashMap<Rule, Severity>,
}

impl Default for LinterSettings {
    fn default() -> Self {
        Self {
            rules: Self::default_rules(),
            severity_overrides: FxHashMap::default(),
        }
    }
}

impl LinterSettings {
    /// Default enabled set: all Correctness and Security rules.
    fn default_rules() -> RuleSet {
        let mut set = RuleSet::EMPTY;
        for rule in Rule::iter() {
            match rule.category() {
                Category::Correctness | Category::Security => set.insert(rule),
                _ => {}
            }
        }
        set
    }

    /// Build settings from user-provided select/ignore lists.
    pub fn from_selectors(
        select: &[RuleSelector],
        ignore: &[RuleSelector],
    ) -> Self {
        let mut rules = RuleSet::EMPTY;
        for sel in select {
            rules = rules.union(&sel.into_rule_set());
        }
        for ign in ignore {
            rules = rules.subtract(&ign.into_rule_set());
        }
        Self { rules, ..Default::default() }
    }
}
```

### Public API

The top-level entry point consumed by the CLI:

```rust
/// Lint an existing parse result and return diagnostics.
pub fn lint_file(
    parse_result: &ParseResult,
    source: &str,
    indexer: &Indexer,
    settings: &LinterSettings,
    suppression_index: Option<&SuppressionIndex>,
    source_path: Option<&Path>,
) -> Vec<Diagnostic> {
    let analysis = analyze_file_at_path(
        &parse_result.file,
        source,
        indexer,
        settings,
        source_path,
    );
    let mut diagnostics = analysis.diagnostics;

    // Add parse-aware rule diagnostics, then apply severity overrides,
    // suppression filtering, per-file ignores, and stable ordering.
    diagnostics.extend(collect_parse_rule_diagnostics(parse_result, source, settings));
    for diag in &mut diagnostics {
        if let Some(&severity) = settings.severity_overrides.get(&diag.rule) {
            diag.severity = severity;
        }
    }
    diagnostics
}
```

### Data Flow (End-to-End)

How the linter fits into `shuck check`:

```
Source text
    → shuck-parser::Parser::parse()       → ParseOutput (Script + Comments)
    → shuck-indexer::Indexer::new()        → Indexer
    → shuck-linter::lint_file()
                                             → Vec<Diagnostic>
    → CLI formats and prints diagnostics
```

The CLI owns the pipeline. The linter is a pure function over parsed input, source text, an indexer, and settings. It performs no I/O, caching, or output formatting.

## Alternatives Considered

### Alternative A: Visitor-Based Dispatch (ruff's Approach)

Walk the AST with a `Visitor` trait, dispatching rules at each node type via `match` arms. This is ruff's primary pattern.

**Rejected as the public rule-authoring model:** Shell lint rules predominantly need semantic information (scoping, use-def chains, declaration classification) rather than raw AST structure. With the semantic model providing pre-computed answers, making every rule visitor-driven would over-couple rules to tree structure. Structural rules that genuinely need AST access can still get it through `checker.ast()`, and inline traversal hooks exist only as an internal seam.

### Alternative B: Event-Based Dispatch

Rules register interest in semantic events (e.g., "on binding created", "on reference resolved") and the semantic model builder fires callbacks during construction.

**Adopted in a limited internal form:** We do expose a non-stable observer seam so the linter can share the semantic traversal for inline checks that depend on evaluation order. The seam remains intentionally narrow and internal; whole-file checks still run after traversal over the completed `SemanticModel`, which preserves testability and keeps most rules query-based.

### Alternative C: Single Flat Numbering (SH-NNN)

Keep the existing `SH-001` through `SH-351` numbering from the Go frontend.

**Rejected because:** Flat numbering provides no structure for prefix-based selection. A user who wants "all style rules" would need to list every code individually. Category prefixes make rule selection ergonomic and give rule codes a semantic meaning visible in output.

### Alternative D: Multi-Letter Prefixes (SHC, SHS, SHP, ...)

Use `SH`-prefixed category codes like `SHC001`, `SHS001`, `SHP001`.

**Rejected because:** The `SH` prefix is redundant — every rule in shuck is a shell rule. Ruff uses terse single-letter prefixes (`E`, `F`, `B`) and they're unambiguous within the tool's context. Single letters are easier to type, read in diagnostic output, and remember. If shuck ever needs to disambiguate from another tool's codes, the tool name in output is sufficient.

## Verification

Once implemented, verify with:

- **Rule registration**: `Rule::iter().count()` returns the expected number of rules, and every rule has a unique code.
- **Prefix selection**: `RuleSelector::from_str("C").unwrap().into_rule_set()` contains exactly the correctness rules. Same for other prefixes.
- **Diagnostic output**: `lint_file()` on a script with a known undefined variable produces a diagnostic with `rule == Rule::UndefinedVariable`, `severity == Severity::Error`, and the correct span.
- **Rule gating**: A `LinterSettings` with only `C001` enabled produces diagnostics only for that rule, even on a script that would trigger others.
- **Severity override**: Setting `severity_overrides[Rule::UnquotedExpansion] = Severity::Hint` produces a diagnostic with `Severity::Hint`.
- **Sort order**: Diagnostics are returned sorted by source position.
- **No-op on empty rule set**: `lint_file()` with `RuleSet::EMPTY` returns an empty vec without running any checks.
