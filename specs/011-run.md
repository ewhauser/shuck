# 011: Shell Script Runner (`shuck run`)

## Status

Implemented

## Summary

Add a `shuck run` subcommand that executes shell scripts with managed interpreter versions, analogous to `uv run` for Python. Users declare which shell and version a script requires — via config file, or CLI flags — and `shuck run` ensures the correct interpreter is available before execution. Prebuilt shell binaries are downloaded on demand and cached globally in `~/.shuck/shells/`. Companion `shuck install` and `shuck shell` commands round out the interface.

## Motivation

Shell scripts are the most widely deployed form of automation, yet they have no standard story for interpreter version management. A script written for bash 5.2 features (like `${var@U}`) silently breaks on macOS's bundled bash 3.2. A zsh completion plugin tested on 5.9 may hit parsing changes in 5.8. Teams paper over this with "works on my machine" or Docker, neither of which is lightweight.

The Python ecosystem solved this with tools like `uv` and `pyenv`. Shell scripts deserve the same treatment:

- **Reproducibility.** A script declares `bash = "5.2"` and every machine runs the same interpreter, whether it's a CI runner, a teammate's laptop, or a fresh install.
- **Testing across versions.** Run a script against bash 4.4 and 5.2 to verify compatibility before shipping.
- **Natural fit for shuck.** Shuck already knows how to detect shell dialects from shebangs and extensions. Adding managed execution closes the loop from "lint your script" to "run your script with the right shell."

## Design

### Goals

- Execute shell scripts with a specific shell interpreter at a specific version.
- Download and cache prebuilt shell binaries on demand.
- Support bash, zsh, dash, and mksh as managed interpreters.
- Accept version constraints from `.shuck.toml`, and CLI flags, with clear precedence.
- Provide `shuck install` to pre-populate the cache and `shuck shell` to enter an interactive session.
- Fall back to system interpreters when explicitly requested via `--system`.

### Non-Goals

- Sandboxing or isolation of script execution.
- Managing non-shell dependencies (e.g., `jq`, `curl`, `awk`).
- Building shells from source. Only prebuilt binaries are supported.
- Windows support in the first iteration.
- Virtual environments or per-project interpreter installs.

### CLI Interface

#### `shuck run`

```
shuck run [OPTIONS] [SCRIPT] [-- ARGS...]
```

| Flag | Short | Description |
|---|---|---|
| `--shell <NAME>` | `-s` | Shell interpreter name (`bash`, `zsh`, `dash`, `mksh`) |
| `--shell-version <CONSTRAINT>` | `-V` | Version constraint (e.g., `5.2`, `>=5.1,<6`, `latest`) |
| `--system` | | Use the system-installed interpreter instead of a managed one |
| `--dry-run` | | Resolve and print the interpreter path without executing |
| `--verbose` | `-v` | Show resolution and download progress |

Examples:

```bash
# Run with the version declared in the script or config
shuck run deploy.sh

# Explicit shell and version
shuck run --shell bash --shell-version 5.2 deploy.sh

# Use whatever bash is on $PATH
shuck run --system deploy.sh

# Pass arguments through to the script
shuck run deploy.sh -- --env staging --force

# Read from stdin
echo 'echo hello' | shuck run --shell bash -
```

When no script is provided, `shuck run` reads from stdin. If no shell or version is otherwise specified or inferred, it falls back to the system `bash`.

#### `shuck install`

```
shuck install [OPTIONS] <SHELL> <VERSION>
```

Pre-downloads a shell interpreter into the global cache without running anything. Useful for CI setup steps or pre-warming a machine.

```bash
shuck install bash 5.2
shuck install zsh 5.9
shuck install --list           # Show available shells and versions
shuck install --list bash      # Show available bash versions
```

#### `shuck shell`

```
shuck shell [OPTIONS]
```

Spawns an interactive shell session using a managed interpreter.

```bash
shuck shell --shell bash --shell-version 5.2
shuck shell                    # Uses project config or defaults
```

### Version Resolution

Version constraints follow a simplified semver-like scheme. Shell versioning is not strictly semver (bash uses `MAJOR.MINOR.PATCH` but zsh uses `MAJOR.MINOR`), so the constraint syntax supports both:

| Constraint | Meaning |
|---|---|
| `5.2` | Exact match: 5.2.x (latest patch) |
| `5.2.21` | Exact match: 5.2.21 specifically |
| `>=5.1` | Any version >= 5.1.0 |
| `>=5.1,<6` | Range: at least 5.1.0, below 6.0.0 |
| `latest` | Most recent available version |

When multiple sources specify a shell or version, precedence is (highest to lowest):

1. **CLI flags** (`--shell`, `--shell-version`)
2. **Inline script metadata** (see below)
3. **Project config** (`.shuck.toml`)
4. **Shebang inference** (shell name only, no version)
5. **File extension inference** (shell name only, no version)
6. **Default shell** (`bash`)

If after resolution only a shell name is known but no version, `shuck run` uses `latest` by default.
If neither shell nor version is resolved from any input, `shuck run` falls back to the system `bash` on `$PATH`.

### Inline Script Metadata

Scripts can declare their interpreter requirements using a structured comment block, following the convention established by PEP 723 / `uv run`:

```bash
# /// shuck
# shell = "bash"
# version = ">=5.1"
# [metadata]
# description = "Deploy script for staging"
# ///
```

The block is TOML enclosed between `# /// shuck` and `# ///` markers. Each line is stripped of the leading `# ` before parsing.

Schema:

```toml
shell = "bash"           # Required: interpreter name
version = ">=5.1"        # Optional: version constraint (default: "latest")

[metadata]               # Optional: informational fields
description = "..."
```

The metadata table is informational only — `shuck run` does not act on it, but it provides a standard place for tooling and humans to document what the script does.

Parsing rules:
- The block must appear before any non-comment, non-blank line.
- Multiple `# /// shuck` blocks are an error.
- Unknown keys outside `[metadata]` are an error (fail fast on typos).

### Project Config (`.shuck.toml`)

A new `[run]` section specifies project-wide defaults:

```toml
[run]
shell = "bash"
shell-version = "5.2"

# Per-shell version pins for the project
[run.shells]
bash = "5.2"
zsh = "5.9"
dash = "0.5.12"
mksh = "59c"
```

The `[run]` top-level `shell` and `shell-version` set defaults for scripts that don't declare their own. The `[run.shells]` table pins specific versions per shell — when a script is detected as bash (by shebang or extension), and no more specific constraint exists, the version from `[run.shells.bash]` is used.

### Interpreter Management

#### Global Cache Layout

Managed interpreters live in `~/.shuck/shells/`:

```
~/.shuck/shells/
  index.json                         # Registry metadata cache
  bash/
    5.2.21/
      x86_64-linux/
        bin/bash
      aarch64-darwin/
        bin/bash
    5.1.16/
      ...
  zsh/
    5.9.1/
      ...
  dash/
    ...
  mksh/
    ...
```

Each interpreter is stored under `<shell>/<version>/<platform>/`. The platform triple follows `<arch>-<os>` (e.g., `x86_64-linux`, `aarch64-darwin`).

#### Registry

A registry index maps shell names and versions to download URLs and checksums. The index is fetched from a well-known URL and cached locally as `~/.shuck/shells/index.json`.

```json
{
  "version": 1,
  "shells": {
    "bash": {
      "versions": {
        "5.2.21": {
          "platforms": {
            "x86_64-linux": {
              "url": "https://shells.shuck.dev/bash/5.2.21/x86_64-linux.tar.gz",
              "sha256": "abc123..."
            },
            "aarch64-darwin": {
              "url": "https://shells.shuck.dev/bash/5.2.21/aarch64-darwin.tar.gz",
              "sha256": "def456..."
            }
          }
        }
      }
    }
  }
}
```

The registry is refreshed at most once per day (controlled by a timestamp in the cache). `shuck install --refresh` forces a fresh fetch.

#### Download and Verification

1. Resolve the shell name, version, and current platform triple.
2. Look up the download URL and expected SHA-256 in the registry index.
3. If the target directory already exists and matches, skip download.
4. Download the archive to a temporary location.
5. Verify SHA-256 before extraction.
6. Extract to a temporary directory under `~/.shuck/shells/`.
7. Atomic rename into the final path.
8. Verify the binary is executable (`<path>/bin/<shell> --version`).

Steps 6-7 ensure that concurrent `shuck run` invocations don't see a half-extracted interpreter.

#### Platform Detection

The current platform is detected at runtime:

| `std::env::consts` | Platform triple |
|---|---|
| `os=linux, arch=x86_64` | `x86_64-linux` |
| `os=linux, arch=aarch64` | `aarch64-linux` |
| `os=macos, arch=x86_64` | `x86_64-darwin` |
| `os=macos, arch=aarch64` | `aarch64-darwin` |

Unsupported platforms produce a clear error with the detected triple.

### System Interpreter Fallback

When `--system` is passed:

1. Skip all managed interpreter resolution.
2. Locate the shell on `$PATH` using `which <shell>`.
3. If a version constraint exists (from metadata or config), query the system shell's version and verify it satisfies the constraint. Fail with a clear error if it doesn't.
4. Execute with the system interpreter.

This is useful in environments where shells are managed externally (NixOS, Homebrew, corporate images) and users just want shuck to verify the version is adequate.

### Execution

Once the interpreter path is resolved:

1. Build the command: `<interpreter_path> <script_path> [args...]`
2. Set up the environment:
   - `SHUCK_SHELL` — the resolved shell name
   - `SHUCK_SHELL_VERSION` — the resolved version
   - `SHUCK_SHELL_PATH` — the interpreter binary path
   - All existing environment variables are inherited.
3. `exec` into the interpreter (replace the shuck process), preserving signal handling, stdin/stdout/stderr, and exit code passthrough.

Using `exec` rather than spawning a child process means the script's PID is the shell's PID, which matters for process management, signal handling, and tools like `pgrep`.

### Integration with Existing Commands

The version resolution and interpreter management machinery is implemented as a shared module so it can be reused:

- `shuck check --shell-version 5.2` — parse using the grammar rules for that shell version (future, out of scope for this spec).
- `shuck format` — no immediate integration, but the plumbing is available.

The `shuck run` command does **not** lint or format before execution. It is purely an interpreter manager and executor. Users who want lint-then-run can compose: `shuck check deploy.sh && shuck run deploy.sh`.

### New Crate: `shuck-run`

The interpreter management logic lives in a new workspace crate `crates/shuck-run/` to keep the main `shuck` binary thin. The crate exposes:

```rust
/// Resolved interpreter ready for execution.
pub struct ResolvedInterpreter {
    pub shell: Shell,
    pub version: Version,
    pub path: PathBuf,
    pub source: ResolutionSource,
}

/// How the interpreter was resolved.
pub enum ResolutionSource {
    Managed,    // Downloaded and cached
    System,     // Found on $PATH
}

/// Shell interpreter identity.
pub enum Shell {
    Bash,
    Zsh,
    Dash,
    Mksh,
}

/// Resolve an interpreter from the given constraints.
pub fn resolve(
    shell: Option<Shell>,
    version: Option<VersionConstraint>,
    system: bool,
    script: Option<&Path>,
    config: Option<&RunConfig>,
) -> Result<ResolvedInterpreter>;

/// Download and cache an interpreter.
pub fn install(
    shell: Shell,
    version: &VersionConstraint,
) -> Result<ResolvedInterpreter>;

/// List available versions from the registry.
pub fn list_available(
    shell: Option<Shell>,
) -> Result<Vec<AvailableShell>>;
```

### Error Handling

| Scenario | Behavior |
|---|---|
| No shell can be inferred | Error: "Cannot determine shell. Specify --shell or add a shebang." |
| Version not available in registry | Error: "bash 5.99 is not available. Run `shuck install --list bash` to see available versions." |
| Platform not available for version | Error: "bash 5.2 does not have a prebuilt binary for x86_64-freebsd." |
| Download fails | Error with HTTP status. Retry once on transient errors (5xx, timeout). |
| SHA-256 mismatch | Error: "Checksum mismatch for bash 5.2.21. Expected abc123, got def456. The archive may be corrupted." Delete partial download. |
| System shell missing | Error: "bash not found on $PATH. Install it or remove --system." |
| System shell version mismatch | Error: "System bash is 3.2.57, but >=5.1 is required. Run `shuck install bash 5.2` to get a managed version." |
| Script not found | Error: "deploy.sh: No such file" |
| Script not executable (system) | Not required — shuck passes the script as an argument to the interpreter. |

## Alternatives Considered

### Use Nix or Docker Instead of Prebuilt Binaries

Nix provides hermetic shell builds and Docker provides full isolation. Both are heavier dependencies than most shell script users want. Nix requires learning a new ecosystem. Docker requires a daemon and adds cold-start overhead. Prebuilt binaries are the lightest option that still provides reproducibility — download once, run instantly. Users who want Nix or Docker can continue using them; `shuck run --system` composes well with both.

### Per-Project Interpreter Installs (Like `node_modules`)

A `.shuck/shells/` per project would provide stronger isolation but shells are large binaries (5-20 MB each) and most projects use the same shell version. A global cache with content-addressed storage avoids redundant downloads across projects. If per-project pinning becomes necessary, the `[run.shells]` config already pins versions — the interpreters just live in the global cache.

### Build from Source Instead of Prebuilt Binaries

Building from source would support every platform and version combination, but it requires a C toolchain, adds minutes of build time, and introduces reproducibility problems from host compiler differences. Prebuilt binaries are faster and more predictable. The trade-off is that we must maintain a build matrix and some niche platforms won't be supported. This is acceptable — `--system` is the escape hatch.

### Use asdf/mise Instead of a Custom Solution

Existing version managers like `asdf` and `mise` already manage shell interpreter versions. However, they require separate installation and configuration. Integrating interpreter management directly into shuck provides a single-tool experience: lint, format, and run with one CLI. The inline metadata format also doesn't exist in other tools — `uv` proved that embedding requirements in the script itself is a significant UX improvement over external configuration.

### Version Pinning via Lock File

A `shuck.lock` file could pin exact versions and checksums, similar to `uv.lock`. This adds complexity for a first iteration where the version constraint in config or script metadata is sufficient. The registry's SHA-256 checksums already provide integrity verification. A lock file may be worth adding later if teams need exact-version coordination beyond what `[run.shells]` provides.

## Security Considerations

- **Download integrity.** All archives are verified against SHA-256 checksums from the registry index before extraction. A compromised CDN cannot inject a malicious binary without also compromising the index.
- **Registry trust.** The registry index is fetched over HTTPS. TLS certificate validation is mandatory. A compromised registry could point to malicious binaries, but the checksums would need to match, limiting the attack to a registry-level compromise.
- **Atomic installs.** Extract-then-rename prevents TOCTOU races where a concurrent process uses a half-extracted interpreter.
- **No implicit execution.** `shuck run` does not auto-execute scripts — it requires an explicit script argument or stdin. There is no "auto-run on file save" or similar ambient execution.
- **Environment passthrough.** All environment variables are inherited. This is intentional (scripts need their environment) but users should be aware that managed interpreters see the same `PATH`, secrets, and credentials as the parent process.

## Verification

### CLI Behavior

```bash
# Resolves interpreter without executing
shuck run --dry-run deploy.sh
# Output: bash 5.2.21 (~/.shuck/shells/bash/5.2.21/aarch64-darwin/bin/bash)

# Downloads and caches on first run
shuck run --shell bash --shell-version 5.2 -c 'echo $BASH_VERSION'
# Output: 5.2.21(1)-release

# System fallback
shuck run --system --shell bash -c 'echo $BASH_VERSION'
# Output: (whatever the system bash reports)

# Version mismatch error
shuck run --system --shell-version '>=5.1' deploy.sh
# Error on macOS: System bash is 3.2.57, but >=5.1 is required.
```

### Cache Integrity

```bash
# Install and verify cache structure
shuck install bash 5.2
ls ~/.shuck/shells/bash/5.2.*/$(uname -m)-$(uname -s | tr A-Z a-z)/bin/bash
# File exists and is executable

# Second install is a no-op
shuck install bash 5.2
# Output: bash 5.2.21 is already installed.
```

### Config Integration

```toml
# .shuck.toml
[run.shells]
bash = "5.2"
```

```bash
# Script without metadata uses project config
shuck run --dry-run deploy.sh
# Resolves to bash 5.2.x from config
```

### Unit Tests

- `resolve()` with CLI flags overrides all other sources.
- `resolve()` with config overrides shebang inference.
- `resolve()` with no constraints and a `#!/bin/bash` shebang resolves to bash latest.
- `resolve()` with `--system` skips the managed cache.
- Version constraint parsing: exact, range, `latest`.
- Platform detection for all supported triples.
- SHA-256 verification failure aborts install.
